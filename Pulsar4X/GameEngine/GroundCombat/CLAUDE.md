# Ground Combat ‚Äî Subsystem Reference

The planet-surface war layer: units you build, station in regions, move, and fight with ‚Äî the ground echo of the space fleet/combat systems. Lives in `GameEngine/GroundCombat/`. **New 2026-07-04 (slice 5a).**

> **Read `docs/GROUND-COMBAT-MAP-DESIGN.md` first** ‚Äî the locked design + the sub-slice roadmap (5a‚Äì5i). v1 target is **full tactical** (unit types + terrain + base-coverage + formations + a navigable click-to-place map), shipped as CI-gated sub-slices. This subsystem sits ON the region map (`Galaxy/PlanetRegionsDB` ‚Äî the 4-slice ring per body).

---

## LOCKED PRINCIPLE ‚Äî every buildable is a real building on the ground (2026-07-04)

The developer's rule: *"everything I build on a planet that's selectable in space is represented by an actual building on the planet itself."* The colony economy (`ComponentInstancesDB` installations) and this ground map are **two views of the SAME physical things** ‚Äî nothing is abstract-only. So any new planet buildable must be a component (`CONVENTIONS.md` ¬ß6) AND carry a **region location** so it draws on the planet view. The reconciliation is **done (#5, 2026-07-04):** `GroundInstallations.LocateColonyInstallations` (hooked in `ColonyFactory.CreateFromBlueprint`) drops a colony's installations into its capital region's `Region.InstallationIds`, so the planet view draws them and fortification counts them. (Follow-up: locate DYNAMICALLY-built installations ‚Äî those added post-creation via the colony economy ‚Äî at their install site; v1 covers the start colony's buildings.) Full rationale + slice mapping: `docs/GROUND-COMBAT-MAP-DESIGN.md` ‚Üí "LOCKED PRINCIPLE." This is *why* the infrastructure system exists: a located building can be bombarded, invaded, defended, and lost region-by-region; an abstract one can't be fought over.

## The model in one line

A planet body carries a **`GroundForcesDB`** (a roster of `GroundUnit`s). Each `GroundUnit` knows its **region**, its **owner faction**, and its **combat stats** ‚Äî so one roster holds both sides of a contested world. Units are **built** at a colony through the normal industry rails (`GroundUnitDesign : IConstructableDesign`), **moved** region‚Üíregion on the region graph's crossing-time edges (5b), **fought** by a strength-math resolver mirroring the space `AutoResolve` (5c), and **captured** by flipping `FactionOwnerID` (5d) ‚Äî the same primitive as a ship/fleet.

## Formations ‚Äî the ground echo of a fleet (5h, 2026-07-04)

A **`GroundFormation`** is a named grouping of units that move and fight as one ‚Äî the ground echo of a `FleetDB`, mirroring its **shape** within the data-object model (the developer's call: *"mirror the fleet as much as possible"*; units aren't entities in v1, so a formation is a serializable data object too). The parity map:

| Fleet | Formation |
|-------|-----------|
| `FleetDB` (entity + `TreeHierarchyDB`) | `GroundFormation` (data object in `GroundForcesDB.Formations`) |
| ship's parent-fleet id (membership on the ship) | `GroundUnit.FormationId` (membership on the unit) |
| `FleetDB.FlagShipID` | `GroundFormation.LeaderUnitId` (references a `GroundUnit.UnitId` ‚Äî a new stable id, the ground echo of an entity id) |

## Why units are DATA objects, not entities (v1)

**Environmental gear (E4) ‚Äî the cradle-to-grave hazard counter.** A `GroundUnitDesign` carries an `EnvironmentalResistance` map (`HazardEffectType` ‚Üí 0..1 fraction negated); `RaiseUnit` snapshots it onto the `GroundUnit.EnvResistance` (like the combat stats), and the attrition step reduces incoming hazard damage by `unit.ResistanceTo(effect)`. This is the ground echo of a ship's `HazardResistanceAtb` ‚Äî a heat-shielded design (`{HeatDamage:0.8}`) outlasts fire that kills an unprotected unit. **v1 folds the "gear" into the design** (its cost rides `ResourceCosts`); making gear a swappable, researched, *installable/strippable* component (the true grave rung) is the v2 promotion, alongside units-as-entities. Gauge: `GroundForcesTests.EnvironmentalGear_ReducesHazardAttrition` (0.8 ‚Üí 20% attrition, 1.0 ‚Üí immune, survives a clone).

A `GroundUnit` is a plain serializable object inside `GroundForcesDB`, not a full `Entity` ‚Äî like `Galaxy.RegionFeature` inside `PlanetRegionsDB`. So a garrison of many units stays cheap and save-safe, and the combat resolver (5c) will read stats directly off the object. The trade-off: it can't reuse the Entity-based `AutoResolve.Resolve` verbatim ‚Äî 5c **mirrors** that salvo-loop math over `GroundUnit`s instead. (If per-unit entities are ever needed ‚Äî commanders, per-unit orders ‚Äî that's a v2 promotion.)

## Connections (Prime Directive)

- **Galaxy / `PlanetRegionsDB`** ‚Äî the surface `GroundForcesDB` sits on (regions = where units stand/move). A unit's `RegionIndex` indexes into `Regions`.
- **Industry / `IConstructableDesign`** ‚Äî `GroundUnitDesign` builds on the existing rails (`IndustryTools.ConstructStuff` ‚Üí `OnConstructionComplete`). A base-mod JSON template (six-point registration, gotcha #10) is the follow-up that makes it player-buildable in a New Game.
- **Colonies / `ColonyInfoDB.PlanetEntity`** ‚Äî the build hook resolves colony ‚Üí its planet ‚Üí the roster.
- **Combat / `AutoResolve`** ‚Äî the strength-math **shape** slice 5c mirrors for ground battles.
- **Capture** ‚Äî clearing a region's garrison flips `FactionOwnerID` (5d), the "take a planet" moment; reuses the fleet-capture primitive and ties into the live colony-damage path (orbital bombardment softens a garrison).
- **Client / `PlanetViewWindow`** ‚Äî 5e (built 2026-07-04) draws units in their region (faction+type tokens with health), click-to-move (`GroundForces.OrderMove`), click-to-place a base (`PlaceInstallationInRegionOrder`), and shows terrain class + `PlanetEnvironmentsDB` hazards ‚Äî the map's upgrade from readout to navigable surface. Runtime is the developer's local build (CI-blind).

## Gotchas

1. **`GroundForcesDB` is on the PLANET BODY, not the colony** ‚Äî parallel to `PlanetRegionsDB`. Forces are *of the planet* so an unowned world can hold a defending garrison. Don't attach it to the colony entity.
2. **Unit stats are a build-time SNAPSHOT on the `GroundUnit`** (not looked up from the design each time) ‚Äî the same choice `ShipCombatValueDB` makes. Changing a design later does not retroactively change units already raised.
3. **`OnConstructionComplete` must never throw** ‚Äî it runs in the daily industry hotloop (a throw there crashes the sim, GameEngine gotcha #1). It's guarded on colony/planet/region-layer/production-line presence.
4. **v1 units are data objects, not entities** ‚Äî so `Entity.Destroy()` / the Entity-based `AutoResolve` don't apply; the 5c resolver operates on `GroundUnit`s. Don't reach for the ship resolver directly.
5. **The Bunker is a real base-mod installation ‚Äî the SIX-point registration (gotcha #10) applies.** `GroundDefenseAtb` binds through the JSON chain: `bunker` ComponentTemplate in `installations.json` (`AtbConstrArgs(PropertyValue('LocalFortify'), PropertyValue('AdjacentProjection'))` ‚Üí `Pulsar4X.GroundCombat.GroundDefenseAtb`, ctor-arg ORDER must match `GroundDefenseAtb(localFortify, adjacentProjection)`; `MountType: PlanetInstallation`) ‚Üí `default-design-bunker` in `componentDesigns.json` ‚Üí `bunker` in `earth.json` `StartingItems` **and** `default-design-bunker` in `earth.json` `ComponentDesigns`. Miss any and New Game / `CreateWithColony` crashes. Gauges: `BaseModIntegrityTests` (builds) + `Fortification_BaseModBunker_BuildsAndWinsTheFight` (the atb binds + wins). Add new ground-defence buildings the same way; new fortification VALUES are per-design data (moddable), not engine constants.
