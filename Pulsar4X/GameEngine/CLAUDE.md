# GameEngine — Subsystem Reference

The `GameEngine` project is the core domain library. It has no dependency on any UI framework. All game logic lives here.

## Core ECS Infrastructure (`Engine/`)

### Entity (`Engine/Entities/Entity.cs`)
- `int Id` — globally unique integer (generated by `EntityIDGenerator`)
- `int FactionOwnerID` — which faction owns this entity (`Game.NeutralFactionId = -99` for neutral)
- `EntityManager? Manager` — the manager this entity belongs to (null if not yet added)
- All DataBlob access delegates to `Manager.GetDataBlob<T>(Id)` etc.
- `Entity.Create()` — static factory, generates a new ID
- `Entity.InvalidEntity` — sentinel with `Id = -1`
- `entity.Destroy()` → `Manager.TagEntityForRemoval(entity)` — actual removal at start of next subpulse

### EntityManager (`Engine/Entities/EntityManager.cs`)
- `SafeDictionary<int, Entity> _entities` — all entities
- `SafeDictionary<Type, SafeDictionary<int, BaseDataBlob>> _datablobStores` — blob type → (entityId → blob)
- `AddEntity(entity, blobs)` — async void, publishes `MessageTypes.EntityAdded`
- `SetDataBlob<T>(entityId, blob)` — async void, publishes `MessageTypes.DBAdded`, triggers `ManagerSubPulse.AddSystemInterupt(blob)` to schedule the blob's processor
- `GetAllEntitiesWithDataBlob<T>()` — primary query for processors
- `GetFilteredEntities(filter, factionId)` — faction-aware query with EntityFilter flags (Friendly/Neutral/Hostile)
- Thread-safety: `SafeDictionary` wraps a plain `Dictionary` with a **`Monitor` (`lock(_lock)`)** — NOT a `ReaderWriterLockSlim` (that claim was stale; corrected 2026-07-18). Two consequences worth knowing before you touch entity/blob-store reads on the parallel sim loop: (1) its `ItemAdded`/`ItemRemoved`/`OnChange` events are raised **after** the lock is released (copy-then-notify) so a subscriber can't deadlock or run arbitrary code under the lock — there are currently zero subscribers, but the contract holds if you add one; (2) the `.Values`/`.Keys` getters return the **live** `Dictionary.ValueCollection`/`KeyCollection` and release the lock *before* the caller enumerates them, so enumerating them (`foreach`/LINQ/`ToList`/`List` ctor) off the writer's thread races a concurrent `Add`/`Remove` and throws "collection was modified." Use **`ValuesSnapshot`/`KeysSnapshot`** (a `List<T>` copied under the lock) for any read that can run concurrently with a mutation. `EntityManager`'s entity/blob-store queries (`GetAllEntitiesWithDataBlob`, `GetFilteredEntities`, `GetAllDataBlobsOfType`, `GetAllEntites`, `GetFirstEntityWithDataBlob`) all use the snapshots for exactly this reason.

### BaseDataBlob (`Engine/Datablobs/BaseDataBlob.cs`)
- Abstract base for all DataBlobs.
- `Entity? OwningEntity` — back-reference to the entity that holds this blob.
- `OnSetToEntity()` — called when blob is added to entity; override to hook in.
- `OnRemovedFromEntity()` — called when blob is removed.
- `Clone()` — must be implemented; used when transferring entities between managers.
- `GetDependencies()` — optional static method returning `List<Type>` of required co-blobs. Used by `EntityManager.AddEntity` validation.

### StarSystem (`Engine/Entities/StarSystem.cs`)
- Extends `EntityManager`.
- Represents one solar system and all entities within it.
- `ManagerSubpulses` — the per-system scheduler.
- `RNG` — seeded PRNG per system for deterministic randomness.

### MasterTimePulse (`Engine/MasterTimePulse.cs`)
- `GameGlobalDateTime` — current game time (fires `GameGlobalDateChangedEvent` on update).
- `Ticklength` — default 1 hour game-time per tick.
- `TimeMultiplier` — wall-clock speed multiplier (default 1.0).
- `StartTime()` / `PauseTime()` / `TimeStep()` — UI-facing controls.
- `DoProcessing(targetDateTime)` — inner loop, processes systems in parallel via `Task.Parallel.ForEach`.
- `AddSystemInteractionInterupt()` — schedules cross-system events (e.g., jumps).
- **Combat interrupt:** `RequestCombatHalt()` (called by the combat engine when a new battle starts) sets `CombatInterruptPending` and cancels the sim — same mechanism as `PauseTime`. The cancel is only honoured at the master-loop boundary, so `SimulateTimeUntil` advances in fine `CombatReactionStep` (5 s) sub-steps instead of a full `Ticklength` **only while a brand-NEW engagement is about to fire** (gated by `AnyNewEngagementImminent()` → `Combat.CombatEngagement.NewEngagementImminent`), letting the clock stop at first contact. An ONGOING fight and normal fast-forward both keep full Ticklength steps, so **combat runs at the player's set speed** — the engine only fine-steps the instant a fight is born, to land the auto-pause. Full rationale: `Combat/CLAUDE.md` → "Combat interrupt — stop the clock at first contact." **FALSE-POSITIVE cap (2026-07-16, from the developer's SIM-STALL log):** `AnyNewEngagementImminent()` can read "imminent" for a hostile pair that never actually engages (a marginal/flickering detection), which used to fine-step at 5 s **forever** → game-time crawled to a standstill while `SensorScan` ran per-sub-step (the "PERF freeze," not a wedge). `SimulateTimeUntil` now counts consecutive fine-steps and, after `MaxConsecutiveFineSteps` (24 = 2 game-minutes) with the loop never cancelled by a real engagement, **stops capping for the rest of that advance** (runs full Ticklength). A GENUINE new engagement cancels the loop within a step or two (`EnsureInCombat` → `RequestCombatHalt`), so it still lands its auto-pause; only the false positive is bounded. **Fixed properly (2026-07-16 round 2 — the developer re-hit it): the cap now PERSISTS across `SimulateTimeUntil` calls.** It was a per-call reset, but in continuous RUNNING the outer loop (`SimulateTimeAsync`) calls `SimulateTimeUntil(now + Ticklength)` once per game-hour, so the reset re-paid the full 24-fine-step tax (24 × a full multi-system `SensorScan`) EVERY hour while a standing imminent read stood — the chronic crawl. The give-up state (`_consecutiveFineSteps`/`_fineStepGivenUp`) is now instance fields, cleared only when nothing is imminent (the episode boundary), so a persistent false positive costs one 24-step burst total and a genuine later engagement — which resolves and stops reading imminent — still re-arms and lands its pause. Paired with the real root-cause fix in `Combat/CombatEngagement.EnsureInCombat` (it seeded the resolver's closing gap to the 1000 km default while the engage gate judged real distance → an engage→freeze→release thrash that kept `NewEngagementImminent` true forever; it now seeds the real `FleetSeparation`). Gauge: `CombatFleetTreeSafetyTests.StandingHostilePair_DoesNotFineStepUnboundedlyAcrossAdvances` + `MasterTimePulse.FineStepCount`.

### ManagerSubPulse (`Engine/ManagerSubPulse.cs`)
- `HotLoopProcessorsNextRun: SafeDictionary<Type, DateTime?>` — when each HotLoop processor next fires.
- `InstanceProcessorsQueue: TimeQueue<(string, Entity)>` — scheduled instance processors (a time-indexed queue of processor-name + entity pairs; the getter returns a defensive copy).
- `ProcessSystem(targetDateTime)` — runs all processors for this system up to the target time.
- `AddEntityInterupt(datetime, processorTypeName, entity)` — schedules a specific entity for instance processing.
- `AddSystemInterupt(DateTime, dbType)` — schedules next HotLoop run for a DataBlob type.
- `AddSystemInterupt(BaseDataBlob)` — called automatically by `EntityManager.SetDataBlob()`.

### ProcessorManager (`Engine/ProcessManager.cs`)
- On construction, uses reflection to find all `IHotloopProcessor` and `IInstanceProcessor` implementations in the assembly.
- `HotloopProcessors: Dictionary<Type, IHotloopProcessor>` — keyed by DataBlob type.
- `_instanceProcessors: Dictionary<string, IInstanceProcessor>` — keyed by `TypeName` property.
- New processors activate automatically by being in the assembly — no manual registration.

### IHotloopProcessor (`Engine/Processors/IHotloopProcessor.cs`)
```csharp
void Init(Game game);
void ProcessEntity(Entity entity, int deltaSeconds);
int ProcessManager(EntityManager manager, int deltaSeconds); // returns entity count processed
TimeSpan RunFrequency { get; }
TimeSpan FirstRunOffset { get; }
Type GetParameterType { get; }  // the DataBlob type this processor handles
```

### IInstanceProcessor (`Engine/Processors/IInstanceProcessor.cs`)
Despite the `I` prefix, this is an **abstract class**, not an interface. Its members are `internal`:
```csharp
internal abstract void ProcessEntity(Entity entity, DateTime atDateTime);
internal string TypeName { get { return GetType().Name; } }  // derived, read-only; key used in InstanceProcessorsQueue
```

### Orders System (`Engine/Orders/`)
- `EntityCommand` — base class for all player orders.
- `OrderableDB` — DataBlob marking an entity as orderable.
- `OrderableProcessor` — `IInstanceProcessor` that executes pending commands.
- `ConditionalOrder` — order that only executes when a condition is met.
- `StandAloneOrderHandler` — the current order handler; validates and enqueues orders. Also enforces the **combat engagement lock**: after `IsValidCommand`, it refuses any order whose commanding entity is a fleet currently in a battle (has a `Combat.FleetCombatStateDB`) unless the order opts in via `EntityCommand.IsAllowedDuringEngagement`. Doctrine changes use a direct call (not an order), so they stay available mid-battle. See `Combat/CLAUDE.md` → "Engagement lock."
- `EntityCommand.IsAllowedDuringEngagement` — virtual, default false; an order overrides it to remain issuable to a fleet locked in an engagement.

## Modding System (`Engine/Modding/`)

- `ModDataStore` — runtime store of all loaded blueprint data.
- `ModLoader` — loads JSON files from the mod data directories. The repo's source-of-truth data lives at `Pulsar4X/GameData/basemod/` (copied to `%AppData%\...\Pulsar4X\Mods\` at client build); there is no `GameEngine/Data/` directory.
- `ModManifest` — mod metadata (name, version, dependencies).
- Blueprints in `Engine/Blueprints/` are the data classes loaded from JSON.
- Active mod data is in `Game.StartingGameData` (a `ModDataStore`).

## Gotchas

1. **Processor auto-discovery means startup failures are silent crashes.** If a new `IHotloopProcessor` has a bad constructor, the game crashes at startup with `NullReferenceException` from `Activator.CreateInstance`. Always keep constructors trivial.

2. **`async void` on EntityManager mutations.** `AddEntity`, `SetDataBlob`, `TagEntityForRemoval`, `RemoveDatablob` are `async void`. Exceptions propagate to the thread pool unobserved. Wrap test assertions around entity mutations carefully.

3. **`ManagerSubPulse.AddSystemInterupt(blob)` uses `_processToDateTime` not `StarSysDateTime`.** This is intentional to avoid temporal anomaly exceptions when blobs are added during processing. Do not change this without understanding the timing invariants.

4. **`Temporal Anomaly Exception`** — thrown when code attempts to schedule something in the past. This surfaces genuine bugs in processor scheduling logic. Never suppress it.

5. **Hotloop processors go to sleep on an empty system, and are re-armed by `SetDataBlob`.** In `ManagerSubPulse.ProcessToNextInterupt`, when a hotloop processor's `ProcessManager` returns `count == 0` (no matching entities in that system), its `HotLoopProcessorsNextRun[type]` is set to `null` — it stops being scheduled, so empty systems don't burn a tick per cycle on empty queries. This is **not** a permanent disable: `EntityManager.SetDataBlob` calls `ManagerSubPulse.AddSystemInterupt(blob)`, which re-arms a `null` entry (it only sets a next-run when the current one is null). So when you colonize/expand into a previously-empty (or previously-processed-empty) system, adding the colony's datablobs wakes mining/industry/etc. back up. **Invariant to preserve:** any future code path that grows an entity into a manager must go through `SetDataBlob` (or otherwise call `AddSystemInterupt`), or that entity's processor will stay asleep. (Looks like a "processor permanently disabled" bug on first read — it isn't, because of the re-arm; confirmed 2026-06-23 while chasing the mining freeze, which was actually a `StarSystem.ActivityState == Stasis` issue, not this.)

6. **`ModLoader.ApplyModGeneric` collection operations tolerate a null base collection.** When a mod uses `CollectionOperation` Add/Remove/Overwrite on a list/dictionary property the *base* blueprint left null, the loader treats it as empty (Add/Overwrite set the mod's collection, Remove is a no-op) rather than throwing. An earlier version threw `NullReferenceException` ("Unable to resolve List/Dictionary for …"), which crashed the whole game at the New Game → Mods page when a stale base blueprint lacked a collection the mod targeted. Two earlier-fixed siblings in the same method: a null/empty `UniqueID` is skipped (illegal dictionary key) **and recorded in `ModLoader.SkippedEntries`** so a silently-dropped base blueprint fails a test instead of hiding (`BaseModIntegrityTests.BaseMod_LoadsWithNoSkippedEntries`), and missing-key lookups elsewhere use `FirstOrDefault`. **Note on data vs. runtime:** the running game loads mods from `%AppData%\Roaming\Pulsar4X\Pulsar4X\Mods\`, which the client build refreshes from the repo's `GameData\**` (`Pulsar4X.Client.csproj` `CopyJsonData` target). The repo's `GameData` and the live `Mods` folder can diverge if a build didn't run or the folder was hand-edited — so prefer robust code over assuming the runtime data matches the repo. Regression test: `ModLoaderTests.TestCollectionOperationOnNullBaseCollectionDoesNotThrow`.
