# GameEngine ‚Äî Subsystem Reference

The `GameEngine` project is the core domain library. It has no dependency on any UI framework. All game logic lives here.

## Core ECS Infrastructure (`Engine/`)

### Entity (`Engine/Entities/Entity.cs`)
- `int Id` ‚Äî globally unique integer (generated by `EntityIDGenerator`)
- `int FactionOwnerID` ‚Äî which faction owns this entity (`Game.NeutralFactionId = -99` for neutral)
- `EntityManager? Manager` ‚Äî the manager this entity belongs to (null if not yet added)
- All DataBlob access delegates to `Manager.GetDataBlob<T>(Id)` etc.
- `Entity.Create()` ‚Äî static factory, generates a new ID
- `Entity.InvalidEntity` ‚Äî sentinel with `Id = -1`
- `entity.Destroy()` ‚Üí `Manager.TagEntityForRemoval(entity)` ‚Äî actual removal at start of next subpulse

### EntityManager (`Engine/Entities/EntityManager.cs`)
- `SafeDictionary<int, Entity> _entities` ‚Äî all entities
- `SafeDictionary<Type, SafeDictionary<int, BaseDataBlob>> _datablobStores` ‚Äî blob type ‚Üí (entityId ‚Üí blob)
- `AddEntity(entity, blobs)` ‚Äî async void, publishes `MessageTypes.EntityAdded`
- `SetDataBlob<T>(entityId, blob)` ‚Äî async void, publishes `MessageTypes.DBAdded`, triggers `ManagerSubPulse.AddSystemInterupt(blob)` to schedule the blob's processor
- `GetAllEntitiesWithDataBlob<T>()` ‚Äî primary query for processors
- `GetFilteredEntities(filter, factionId)` ‚Äî faction-aware query with EntityFilter flags (Friendly/Neutral/Hostile)
- Thread-safety: `SafeDictionary` wraps a `Dictionary` with a `ReaderWriterLockSlim`.

### BaseDataBlob (`Engine/Datablobs/BaseDataBlob.cs`)
- Abstract base for all DataBlobs.
- `Entity? OwningEntity` ‚Äî back-reference to the entity that holds this blob.
- `OnSetToEntity()` ‚Äî called when blob is added to entity; override to hook in.
- `OnRemovedFromEntity()` ‚Äî called when blob is removed.
- `Clone()` ‚Äî must be implemented; used when transferring entities between managers.
- `GetDependencies()` ‚Äî optional static method returning `List<Type>` of required co-blobs. Used by `EntityManager.AddEntity` validation.

### StarSystem (`Engine/Entities/StarSystem.cs`)
- Extends `EntityManager`.
- Represents one solar system and all entities within it.
- `ManagerSubpulses` ‚Äî the per-system scheduler.
- `RNG` ‚Äî seeded PRNG per system for deterministic randomness.

### MasterTimePulse (`Engine/MasterTimePulse.cs`)
- `GameGlobalDateTime` ‚Äî current game time (fires `GameGlobalDateChangedEvent` on update).
- `Ticklength` ‚Äî default 1 hour game-time per tick.
- `TimeMultiplier` ‚Äî wall-clock speed multiplier (default 1.0).
- `StartTime()` / `PauseTime()` / `TimeStep()` ‚Äî UI-facing controls.
- `DoProcessing(targetDateTime)` ‚Äî inner loop, processes systems in parallel via `Task.Parallel.ForEach`.
- `AddSystemInteractionInterupt()` ‚Äî schedules cross-system events (e.g., jumps).
- **Combat interrupt:** `RequestCombatHalt()` (called by the combat engine when a new battle starts) sets `CombatInterruptPending` and cancels the sim ‚Äî same mechanism as `PauseTime`. The cancel is only honoured at the master-loop boundary, so `SimulateTimeUntil` advances in fine `CombatReactionStep` (5 s) sub-steps instead of a full `Ticklength` **when a battle is active/imminent** (gated by `AnyActiveSystemInOrNearCombat()` ‚Üí `Combat.CombatEngagement.CombatActiveOrImminent`), letting the clock stop at first contact. Away from combat it takes full Ticklength steps as before, so fast-forward is unaffected. Full rationale: `Combat/CLAUDE.md` ‚Üí "Combat interrupt ‚Äî stop the clock at first contact."

### ManagerSubPulse (`Engine/ManagerSubPulse.cs`)
- `HotLoopProcessorsNextRun: SafeDictionary<Type, DateTime?>` ‚Äî when each HotLoop processor next fires.
- `InstanceProcessorsQueue: SortedDictionary<DateTime, Dictionary<string, List<Entity>>>` ‚Äî scheduled instance processors.
- `ProcessSystem(targetDateTime)` ‚Äî runs all processors for this system up to the target time.
- `AddEntityInterupt(datetime, processorTypeName, entity)` ‚Äî schedules a specific entity for instance processing.
- `AddSystemInterupt(DateTime, dbType)` ‚Äî schedules next HotLoop run for a DataBlob type.
- `AddSystemInterupt(BaseDataBlob)` ‚Äî called automatically by `EntityManager.SetDataBlob()`.

### ProcessorManager (`Engine/ProcessManager.cs`)
- On construction, uses reflection to find all `IHotloopProcessor` and `IInstanceProcessor` implementations in the assembly.
- `HotloopProcessors: Dictionary<Type, IHotloopProcessor>` ‚Äî keyed by DataBlob type.
- `_instanceProcessors: Dictionary<string, IInstanceProcessor>` ‚Äî keyed by `TypeName` property.
- New processors activate automatically by being in the assembly ‚Äî no manual registration.

### IHotloopProcessor (`Engine/Processors/IHotloopProcessor.cs`)
```csharp
void Init(Game game);
void ProcessEntity(Entity entity, int deltaSeconds);
int ProcessManager(EntityManager manager, int deltaSeconds); // returns entity count processed
TimeSpan RunFrequency { get; }
TimeSpan FirstRunOffset { get; }
Type GetParameterType { get; }  // the DataBlob type this processor handles
```

### IInstanceProcessor (`Engine/Processors/IInstanceProcessor.cs`)
```csharp
void ProcessEntity(Entity entity, DateTime datetime);
string TypeName { get; }  // key used in InstanceProcessorsQueue
```

### Orders System (`Engine/Orders/`)
- `EntityCommand` ‚Äî base class for all player orders.
- `OrderableDB` ‚Äî DataBlob marking an entity as orderable.
- `OrderableProcessor` ‚Äî `IInstanceProcessor` that executes pending commands.
- `ConditionalOrder` ‚Äî order that only executes when a condition is met.
- `StandAloneOrderHandler` ‚Äî the current order handler; validates and enqueues orders. Also enforces the **combat engagement lock**: after `IsValidCommand`, it refuses any order whose commanding entity is a fleet currently in a battle (has a `Combat.FleetCombatStateDB`) unless the order opts in via `EntityCommand.IsAllowedDuringEngagement`. Doctrine changes use a direct call (not an order), so they stay available mid-battle. See `Combat/CLAUDE.md` ‚Üí "Engagement lock."
- `EntityCommand.IsAllowedDuringEngagement` ‚Äî virtual, default false; an order overrides it to remain issuable to a fleet locked in an engagement.

## Modding System (`Engine/Modding/`)

- `ModDataStore` ‚Äî runtime store of all loaded blueprint data.
- `ModLoader` ‚Äî loads JSON files from `Data/` directories.
- `ModManifest` ‚Äî mod metadata (name, version, dependencies).
- Blueprints in `Engine/Blueprints/` are the data classes loaded from JSON.
- Active mod data is in `Game.StartingGameData` (a `ModDataStore`).

## Gotchas

1. **Processor auto-discovery means startup failures are silent crashes.** If a new `IHotloopProcessor` has a bad constructor, the game crashes at startup with `NullReferenceException` from `Activator.CreateInstance`. Always keep constructors trivial.

2. **`async void` on EntityManager mutations.** `AddEntity`, `SetDataBlob`, `TagEntityForRemoval`, `RemoveDatablob` are `async void`. Exceptions propagate to the thread pool unobserved. Wrap test assertions around entity mutations carefully.

3. **`ManagerSubPulse.AddSystemInterupt(blob)` uses `_processToDateTime` not `StarSysDateTime`.** This is intentional to avoid temporal anomaly exceptions when blobs are added during processing. Do not change this without understanding the timing invariants.

4. **`Temporal Anomaly Exception`** ‚Äî thrown when code attempts to schedule something in the past. This surfaces genuine bugs in processor scheduling logic. Never suppress it.

5. **Hotloop processors go to sleep on an empty system, and are re-armed by `SetDataBlob`.** In `ManagerSubPulse.ProcessToNextInterupt`, when a hotloop processor's `ProcessManager` returns `count == 0` (no matching entities in that system), its `HotLoopProcessorsNextRun[type]` is set to `null` ‚Äî it stops being scheduled, so empty systems don't burn a tick per cycle on empty queries. This is **not** a permanent disable: `EntityManager.SetDataBlob` calls `ManagerSubPulse.AddSystemInterupt(blob)`, which re-arms a `null` entry (it only sets a next-run when the current one is null). So when you colonize/expand into a previously-empty (or previously-processed-empty) system, adding the colony's datablobs wakes mining/industry/etc. back up. **Invariant to preserve:** any future code path that grows an entity into a manager must go through `SetDataBlob` (or otherwise call `AddSystemInterupt`), or that entity's processor will stay asleep. (Looks like a "processor permanently disabled" bug on first read ‚Äî it isn't, because of the re-arm; confirmed 2026-06-23 while chasing the mining freeze, which was actually a `StarSystem.ActivityState == Stasis` issue, not this.)

6. **`ModLoader.ApplyModGeneric` collection operations tolerate a null base collection.** When a mod uses `CollectionOperation` Add/Remove/Overwrite on a list/dictionary property the *base* blueprint left null, the loader treats it as empty (Add/Overwrite set the mod's collection, Remove is a no-op) rather than throwing. An earlier version threw `NullReferenceException` ("Unable to resolve List/Dictionary for ‚Ä¶"), which crashed the whole game at the New Game ‚Üí Mods page when a stale base blueprint lacked a collection the mod targeted. Two earlier-fixed siblings in the same method: a null/empty `UniqueID` is skipped (illegal dictionary key) **and recorded in `ModLoader.SkippedEntries`** so a silently-dropped base blueprint fails a test instead of hiding (`BaseModIntegrityTests.BaseMod_LoadsWithNoSkippedEntries`), and missing-key lookups elsewhere use `FirstOrDefault`. **Note on data vs. runtime:** the running game loads mods from `%AppData%\Roaming\Pulsar4X\Pulsar4X\Mods\`, which the client build refreshes from the repo's `GameData\**` (`Pulsar4X.Client.csproj` `CopyJsonData` target). The repo's `GameData` and the live `Mods` folder can diverge if a build didn't run or the folder was hand-edited ‚Äî so prefer robust code over assuming the runtime data matches the repo. Regression test: `ModLoaderTests.TestCollectionOperationOnNullBaseCollectionDoesNotThrow`.
