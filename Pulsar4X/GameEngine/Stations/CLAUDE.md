# Stations ‚Äî Subsystem Reference

Space stations ‚Äî the **cheap, fast, flexible, FRAGILE alternative to a planetary colony**. Lives in `GameEngine/Stations/`. New as of 2026-06-29.

> **Read `docs/economy/OFF-WORLD-INFRASTRUCTURE-DESIGN.md` before touching this.** It holds the locked architecture decision (PARALLEL host, NOT a generalized colony) and the open tuning questions (cost-gradient curve, durability/invasion numbers, manning). This subsystem is the foundation slice of that design.

---

## The one idea

A station does the same off-world jobs a colony does ‚Äî mine, refine, research, trade, house people ‚Äî by carrying the **same component-equipment chassis** a colony carries. But it is its **own host type** so it can own its own cost curve, durability, and invasion math. That distinctness is the whole point: a station is cheap while focused and expensive as a planet-replacement, and a fraction of the effort to destroy compared to a planet's long ground war ("blowing on a flower vs. pushing a Seawolf-class submarine").

**Why parallel and not a planet-less colony:** those two trade-offs only exist if stations and planets are mechanically distinct hosts. See the design doc's "Why PARALLEL" section.

## The shared chassis (what `StationFactory` attaches)

Mirrors `ColonyFactory.CreateColony` exactly, because the economy processors discover their work by these ability blobs, not by host type:

`NameDB`, `StationInfoDB`, `MiningDB`, `OrderableDB`, `MassVolumeDB`, `CargoStorageDB`, `PositionDB` (relative to the hosting body), `TeamsHousedDB`, `ComponentInstancesDB` (installed modules live here), `InfrastructureDB`, `ColonyMoraleDB`, `LegitimacyDB`, `RebellionDB`, `ColonySustenanceDB`, `StationEconomyDB` (the Slice C upkeep key).

**Deliberately NOT attached: `ColonyBonusesDB`.** Its `GetDependencies()` hard-requires `ColonyInfoDB` (which a station doesn't have), so attaching it fails `EntityManager.AddEntity`'s dependency validation (`AreAllDependenciesPresent()` ‚Üí `InvalidOperationException`). How a station carries production/research/mining bonuses is a decision for the economy-wiring slice (task #17) ‚Äî relax that dependency, or give the station a station-compatible bonuses blob. This was caught by CI on the first push (the foundation's first red ‚Üí green).

It is then registered: `factionInfo.Stations.Add(station)` + `FactionOwnerDB.SetOwned(station)`.

## The key finding ‚Äî economy comes for FREE, population does NOT

The economy processors query by **ability component**, not by `ColonyInfoDB`:
- `MineResourcesProcessor` ‚Üí `GetAllEntitiesWithDataBlob<MiningDB>()`
- `IndustryProcessor` ‚Üí `GetAllEntitiesWithDataBlob<IndustryAbilityDB>()` (auto-attached when an industry module is installed)
- `ResearchProcessor` ‚Üí `GetAllEntitiesWithDataBlob<ResearcherDB>()` (spawned by a research-lab component)

So a station that carries the matching components is mined / built / researched **without any station-aware code**. The exceptions:
1. **Mining ‚Äî WIRED 2026-06-30 (task #17, slice 1).** `MineResourcesProcessor` was hard-coded to read `MineralsDB` off `colonyInfoDB.PlanetEntity` in three spots (`ProcessEntity`, `CalcMaxRate`, and `MiningHelper.CalculateActualMiningRates`). All three now resolve the resource body through the host-agnostic helper **`MiningHelper.TryGetMiningBody(entity, out bodyEntity)`** ‚Äî colony ‚Üí `PlanetEntity`, station ‚Üí `HostingBodyEntity` (guards an unset/invalid body so it can't NPE). A station that carries a mine component + a cargo hold now mines its hosting body exactly like a colony. `InfrastructureProcessor.GetEfficiency` and the recalc path were already host-agnostic. Mining bonuses come from `ColonyBonusesDB` (a station has none ‚Üí defaults to √ó1.0). Gauge: `StationFactoryTests.Station_WithMiningModule_MinesItsHostingBody` (a station carrying the colony's own mine + cargo designs accrues `CargoStorageDB.TotalStoredMass`) + `TryGetMiningBody_ResolvesColonyAndStation`.
2. **Population ‚Äî WIRED 2026-06-30 (task #17, population half) via a PARALLEL processor.** `PopulationProcessor` stays hard-keyed to `ColonyInfoDB` (untouched ‚Üí zero colony-regression risk); a new **`StationPopulationProcessor`** keyed on `StationInfoDB` grows/starves a manned station's `Population`. The key difference, and why it's a separate processor not a shared path: a station is a **SEALED HABITAT** ‚Äî its population cap is the life support its habitat modules (`PopulationSupportAtbDB`) provide, **NOT** the body's habitability (`ColonyCost`). So a station with no habitat modules is a **tomb** (population dies off); one with ample habitat grows toward that cap (a native world like Earth grows un-capped ‚Äî a station never does, which is its fragility). The **morale machinery is shared/host-agnostic**: `StationFactory` now attaches `ColonyMoraleDB`, and the processor feeds it station-appropriate inputs (no planet cost; crowding vs the habitat cap; jobs/comfort from the same module extensions). Gauge: `StationFactoryTests.MannedStation_WithNoLifeSupport_LosesPopulation`.

## The grave rung ‚Äî a station is now KILLABLE and LOSE-ABLE (Slice B, 2026-07-03)

The whole reason a station is its OWN host (not a planet-less colony) is the durability asymmetry: it's **cheap to build, cheap to kill** ‚Äî an attacker's quick choice, not a planet's long ground war. Before Slice B a station was a **ghost target**: a weapon could lock and fire on it, but `DamageProcessor.OnTakingDamage` hit no branch for it and **returned 0 damage** ‚Äî invulnerable, and if it *had* been destroyed it would have leaked dangling refs. Slice B closes that:

- **`StationInfoDB.StructuralIntegrity`** ‚Äî a FLAT placeholder durability pool (`BaseStructuralIntegrity = 500`). It is **the placeholder ratio**: a planet has NO kill trigger (`OnColonyDamage` never destroys a colony ‚Üí effectively ‚àû), a station has this modest finite pool ‚Üí "a fraction of the effort to destroy that a planet does." Deliberately **not** scaled by module count ‚Äî a bigger station has more to *lose*, not more to *survive*.
- **`DamageProcessor.OnStationDamage`** (routed from `OnTakingDamage` via a `StationInfoDB` branch) ‚Äî mirrors `OnColonyDamage` (population casualties + random module damage, factored into shared `ApplyPopulationCasualties`/`ApplyInstallationDamage` helpers so the two hosts never drift), but **skips atmospheric contamination** (no atmosphere) and **adds the structural-integrity kill**: drain the pool, and at ‚â§ 0 call `StationFactory.DestroyStation`.
- **`StationFactory.DestroyStation`** ‚Äî the clean teardown (see file map). Critically it tears down **spawned sub-entities**: a research lab's `ResearcherDB` lives on a SEPARATE entity (`ComponentInstance.SpawnedEntityId`) that `RemoveComponentInstance` does NOT clean up ‚Äî so without this a dead research station would keep researching from the grave.

**This wires the DIRECT weapon-hit path** (`BeamWeaponProcessor.OnHit` / `MissileImpactProcessor` ‚Üí `DamageProcessor.OnTakingDamage`) ‚Äî a ship that fires on a station now damages/destroys it. **Follow-on (NOT in Slice B):** the fleet **auto-resolve** engine (`Combat/CombatEngagement`) is `FleetDB`-keyed, so a station is not yet a target *inside a fleet battle* ‚Äî that needs a non-fleet "installation combatant" representation (or a dedicated bombardment order). And **invasion/capture** (taking a station intact rather than destroying it) is unbuilt ‚Äî it rides on ground combat (`docs/aurora/GROUND-COMBAT.md`). Gauges: `StationFactoryTests` ‚Äî `Station_TakesDamage_DrainsStructuralIntegrity_ButSurvivesASmallHit`, `Station_DestroyedWhenStructuralIntegrityExhausted_AndUnregisters`, `DestroyedStation_TearsDownSpawnedResearcher`, `MannedStation_Bombardment_KillsPopulation`.

## The frame's material cost + fleet-pooled supply (Slice F, 2026-07-03)

The last "free" rung: before Slice F the deploy conjured a station from nothing. Now the bare **platform frame costs refined materials** you must carry to the site ‚Äî the materials-supply loop A2 deferred. This closes the station's **cradle-to-grave production rung** (mineral ‚Üí material ‚Üí *frame built from that material* ‚Üí component ‚Üí ‚Ä¶ ‚Üí loss).

- **The cost** lives on `DeployStationOrder`: placeholder consts `FrameMaterialId = "stainless-steel"`, `FrameFractionOfShipCapacity = 0.5`, `FrameFloorUnits = 10`. `FrameCostUnits(ship, material)` = a placeholder fraction of the **construction ship's own cargo capacity** (in the frame material's cargo type), floored ‚Äî so a bigger constructor drops a bigger, costlier platform ("dependent on how it's built"). v1 bills **one** material; a multi-material bill is a refinement. Tune like Combat's `SalvoDamageScale`.
- **The fleet pool** (`GatherPooledHolds`): there is **no ship‚Üífleet back-reference** (confirmed ‚Äî ships carry no fleet field), so the deploy finds the ship's fleet by searching the system's `FleetDB` entities for the one whose `Children` contain the ship, then pools every fleet-mate's `CargoStorageDB` with the ship's own. **Direct children only** ‚Äî sub-fleet recursion (`CombatEngagement.GetFleetShips` is the recursive walk to copy) is a documented refinement. A ship in no fleet just spends its own hold.
- **Check-then-consume** (`TryPayFrameMaterials`): sum the pool's units of the material FIRST; if it can't cover the bill, **refuse** the deploy (no station, a `MineralShortage` event is published) and consume nothing ‚Äî a refused deploy never half-drains a hold. Else drain hold-by-hold (mirrors `IndustryTools.ConsumeResources`, which is the reference consume-from-a-hold loop; `RemoveCargoByUnit` clamps silently, so we subtract the returned `used` each hold).
- **The empty-hold fix (the flagged bug):** a bare `new CargoStorageDB()` has an **empty `TypeStores`**, so *every* `AddCargoByUnit`/`RemoveCargoByUnit` on it **silently returns 0 and no-ops** (all the error logging in `CargoMath` is commented out ‚Äî an invisible trap). A station made by `CreateStation` still has that bare hold (correct ‚Äî deploy-bare, add cargo modules), but the **deploy starter loadout now includes `default-design-warehouse`**, whose `CargoStorageAtb.OnComponentInstallation` seeds a real `general-storage` `TypeStore` ‚Äî so a *deployed* station has a working hold that can receive the materials its in-situ builds need. (To give any other station a hold, install a cargo/`CargoStorageAtb` module; don't rely on the bare blob.)

**Gauges** (`StationFactoryTests`): `ConstructionShip_DeploysStation_AtAStar_AndSurvives` (updated ‚Äî seeds the constructor's hold, asserts frame material consumed + the station has a real `general-storage` store), `ConstructionShip_PoolsFleetMaterials_OnDeploy` (constructor's own hold empty, a fleet-mate freighter supplies the materials ‚Üí deploy succeeds, freighter drained), `ConstructionShip_InsufficientMaterials_RefusesDeploy` (no material anywhere ‚Üí no station). **Client runtime note (CI-blind):** a construction ship now needs `stainless-steel` loaded ‚Äî in its own hold or a fleet-mate's ‚Äî or the deploy is refused with a shortage event; verify the load-then-deploy flow on the local build.

**Deploy tests submit through the REAL order handler ‚Äî the player path (standing convention, 2026-07-03).** All the deploy gauges above submit via `SubmitDeployAsPlayer(s, command)` ‚Üí **`s.Game.OrderHandler.HandleOrder(command)`** ‚Äî the *exact* call the client's "Deploy Station Here" map button makes (`EntityContextMenu` ‚Üí `Game.OrderHandler.HandleOrder`) ‚Äî **not** a direct `command.Execute(...)`. This is deliberate and permanent: it exercises the true player submission path (`IsValidCommand` gate ‚Üí the combat **engagement lock** ‚Üí the **action-lane** queue onto the ship's `OrderableDB` ‚Üí the `OrderableProcessor` running the order), so the test proves what a player actually triggers, not just the raw execution. `DeployStationOrder` is an `InstantOrder` with a default (due) `ActionOnDate`, so the handler runs it **synchronously** ‚Äî assert right after. Caveat: `HandleOrder` *contains + logs* any exception the order throws (the guard the live UI relies on so one bad order can't blank the frame), so a throwing deploy surfaces as a silently-failed order ("no station", trace on the console), not a test-thread crash ‚Äî if a deploy gauge mysteriously shows no station, check the console for `[OrderError]`. **New deploy tests must use `SubmitDeployAsPlayer`, never `command.Execute` directly.**

## STATION-as-assembly ‚Äî build-then-deploy a designed station (2026-07-15)

The station now has the SAME buildable shape a ship / ground unit does: a **`StationDesign`** (chassis + modules) is an `IConstructableDesign` you queue at a colony; when the build completes it DEPLOYS a real station at that colony's body with the designed modules installed (`StationDesign.OnConstructionComplete` ‚Üí `StationFactory.CreateStation` + `AddComponent` per module). This is the parallel front-door to the ship-issued `DeployStationOrder` ‚Äî the *industry* path (build it at home, it appears in orbit) vs. the *construction-ship* path (carry a bare frame to a star). Both end at `StationFactory.CreateStation`.

**The base-mod `station-chassis` is a full SIX-point registration (gotcha #10)** ‚Äî the New-Game-crash gate, kept green by `BaseModIntegrityTests`:

| Rung | Where |
|------|-------|
| template ‚úì | `station-chassis` ComponentTemplate in `TemplateFiles/installations.json` (`stationChassisAtbArgs` ‚Üí `AtbConstrArgs(PropertyValue('Structural Budget'))` ‚Üí `Pulsar4X.Stations.StationChassisAtb`; `MountType: Station`; materials `stainless-steel`/`aluminium`/`iron` ‚Äî all already unlocked) |
| componentDesign ‚úì | `default-design-station-chassis` in `ScenarioFiles/designs/componentDesigns.json` |
| StartingItems ‚úì | `station-chassis` in `earth.json` `StartingItems` (unlocks the TEMPLATE ‚Äî the 6th point) |
| ComponentDesigns ‚úì | `default-design-station-chassis` in `earth.json` `ComponentDesigns` |
| materials-unlocked ‚úì | `stainless-steel` / `aluminium` / `iron` already in `StartingItems` (no new mineral) |

**Station-mount modules:** the assembler filters station parts by `ComponentMountType.Station`. Five existing base-mod module templates gained `Station` in their `MountType` flags (purely additive ‚Äî no new templates, no resolver change): **`reactor`** (energy.json), **`research-lab`**, **`factory`**, **`refinery`**, **`mine`** (installations.json) ‚Äî power / research / production / refining / mining, the sensible station jobs. The `StationDesign` itself is NOT authored in JSON (built at runtime by `RegisterStationDesign`), so only the chassis needs the six-point chain; the assembler (next slice) assembles `StationDesign`s from the chassis + these modules.

Gauge: `StationDesignTests` ‚Äî the chassis binds its atb (Structure budget), `RegisterStationDesign` makes a buildable, and a completed build deploys a station with its modules installed.

## Gotchas

1. **`PositionDB`'s active class is in `Pulsar4X.Movement`, not `Datablobs`** (the Datablobs copy is commented out). `StationFactory` imports `Pulsar4X.Movement` for it ‚Äî the same trap that bit the test harness.

5. **A bare `new CargoStorageDB()` silently no-ops (Slice F).** No `TypeStores` ‚Üí `AddCargoByUnit`/`RemoveCargoByUnit` return 0 with no error (logging commented out). `StationFactory.CreateStation`'s hold is intentionally bare (install a cargo module to use it); the deploy loadout adds a warehouse so a deployed station's hold works. Never assume a fresh `CargoStorageDB` can hold anything.
2. **`FactionInfoDB.Stations` must be added to the FactionInfoDB copy-ctor**, or it's dropped on `Clone()` / save-load round-trip. It is (mirrors the `Colonies` line). Any new faction-level list needs the same treatment. **`StructuralIntegrity` likewise must be in the `StationInfoDB` copy-ctor** (it is) or a damaged station heals to full on save/load.
3. **The player REACHES a station via a CONSTRUCTION SHIP (Slice A‚ÜíA2, 2026-07-03).** Fly a hauler to a spot (a **star**, a belt, a planet orbit) ‚Üí right-click it ‚Üí **"Deploy Station Here"** ‚Üí a platform anchors to the ship's SOI body, carries a starter constructor ‚Üí select it ‚Üí queue modules via the embedded `IndustryDisplay`. The old planet-list "Deploy Station" button was REMOVED ‚Äî it was survey-gated/body-bound and couldn't place a station around a star, which defeated the concept. The **engine order is CI-tested** (`ConstructionShip_DeploysStation_AtAStar_AndSurvives` ‚Äî literally deploys at a star); the **UI is compile-checked only** (CI can't run the client ‚Äî verify the context-menu action + StationWindow live). Remaining front-door polish (consume the constructor/materials from the ship's hold via the `LogiBaseDB` materials loop, a station designer #21) is deferred.
4. **`DestroyStation` must run BEFORE `FactionOwnerDB.RemoveEntity` zeroes `FactionOwnerID`.** It captures the owning faction first, then unregisters ‚Äî reversing the order would lose the faction handle.

## Tests

**The void habitat (data, 2026-06-30).** A manned station needs life-support modules or it starves (the sealed-habitat model). The base game had NO population-support design that works off-planet ‚Äî the only one, `default-design-infrastructure` ("Earth-Standard Infrastructure"), is gravity/pressure-toleranced to Earth (8.8‚Äì10.8 m/s¬≤, 0.9‚Äì1.1 atm), so it provides ZERO support in microgravity. Added a **`space-habitat` ComponentTemplate** (`TemplateFiles/installations.json`) + a **`default-design-space-habitat`** design ("Space Habitat") ‚Äî the same `PopulationSupportAtbDB` + `InfrastructureCapacityAtb` + `CargoStorageAtb` as infrastructure, but with **NO `GravityToleranceAtb`/`PressureToleranceAtb` at all**, so a sealed self-contained habitat supports population on ANY body. (Why a new template, not the infrastructure one with tolerance values set to 0: the template's gravity/pressure values are clamped to the tech-formula bounds at instantiation ‚Äî ~9.81 m/s¬≤ / ~1.0 atm with no tech ‚Äî so an override to 0 gets pulled back to Earth-ish. `GetPopulationSupportValue` only applies a tolerance gate when the attribute is *present*, so omitting it is the clean fix. Caught by CI: the first cut set tolerance values to 0 and the habitat still gave 0 support in microgravity.) Unlocked at start by adding **(a)** the design id `default-design-space-habitat` to the Earth colony blueprint's `ComponentDesigns` list AND **(b)** the *template* id `space-habitat` to its **`StartingItems`** list ‚Äî both in `systems/sol/earth.json`. (Gotcha, cost a CI cycle: a new ComponentTemplate starts in `LockedComponentTemplates`; `ComponentDesignFromJson` looks up the *unlocked* `ComponentTemplates`, so a design whose template isn't in `StartingItems` throws `"<template> was not found in the faction data store"` on EVERY colony build ‚Äî see `Combat/CLAUDE.md` note. Adding the template to `StartingItems` unlocks it.) This is the data that makes an asteroid station viable. Gauge: `StationFactoryTests.SpaceHabitat_SupportsPopulationInMicrogravity_WhereEarthInfraCannot` (Earth infra ‚Üí 0 support at an asteroid, space habitat ‚Üí >0, and a habitat-supported station grows).

**Listening-outpost flavor ‚Äî FREE (host-agnostic, Slice E, 2026-07-03).** A "listening outpost" ‚Äî a station out at a distant star for early-warning/detection coverage ‚Äî is just a station carrying a passive sensor. Detection is host-agnostic: `SensorReceiverAtb.OnComponentInstallation` ‚Üí `SensorTools.SetInstances` gives the station a `SensorAbilityDB`, and `SensorScan` keys on that blob, not on the host type. **The one gap that had to be closed** was the *mid-game first-scan kick*: `SensorScan` self-reschedules but needs a first fire, which at New Game comes from `Game.PostNewGameInitialization` ‚Äî a station built *later* never got it, so it sat deaf. Fixed in `OnComponentInstallation` (schedules the first scan on install ‚Äî see `Sensors/CLAUDE.md` ‚Üí "The first-scan kick"). Data: a bigger **`default-design-outpost-sensor`** ("Deep-Space Listening Array", antenna 12000 off the existing `passive-sensor` template ‚Äî reuses an already-unlocked template, so no gotcha-10 crash) added to the Earth blueprint's `ComponentDesigns`. Gauge: `StationFactoryTests.ListeningOutpostStation_DetectsHostileShip_ViaInstallScheduledScan` (a station carrying the array detects a hostile ship with only the clock advancing ‚Äî proving both host-agnostic detection AND the install-kick). *(A dark sensor station out at a far star for actual early warning is the intended use; placing it there is the deploy path ‚Äî Slice A2 construction ship + the Slice D Lagrange anchors.)*

**Research station flavor ‚Äî FREE (host-agnostic, gauged 2026-06-30, task #18).** A "research station" is just a station carrying a research-lab module ‚Äî no station-specific code at all. `ResearchPointsAtbDB.OnComponentInstallation` spawns a `ResearcherDB` on a new entity tagged with `LocationId = the host` (colony OR station), and `ResearchProcessor` keys on `ResearcherDB`, so the lab researches wherever it's installed (no scientist needed ‚Äî the lab sets `PointsPerDay` directly; cost is billed to faction funds).
