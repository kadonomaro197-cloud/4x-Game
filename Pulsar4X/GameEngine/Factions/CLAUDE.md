# Factions — Subsystem Reference

Faction state, tech database, ability store, and NPC decision engine. Lives in `GameEngine/Factions/`.

---

## File Map

| File | Purpose |
|------|---------|

Gauge: `OpeningDiplomacyTests`. |
| `DoctrineVector.cs` | **NEW** Struct: four float weights (Economic, Military, Tech, Expansion) describing NPC strategic priorities. Serialized with `[JsonProperty]`; lives on `FactionInfoDB.Doctrine`. |
| `NPCDecisionProcessor.cs` | `IHotloopProcessor` on `FactionInfoDB`, monthly (`FirstRunOffset` 5d, `RunFrequency` 30d). Only acts on factions where `IsNPC == true`. **Full decision + planner + diplomacy + espionage engine (510 lines, `NPCDecisionProcessor.cs`), NOT a skeleton.** Each `Tick`: settles a strategic objective (needs-ladder → `ObjectiveSelector` → `ObjectiveTransition` → `StrategicObjectiveDB`), then behind default-off gates runs the means-ends order planner (`EnableOrderEmission`), treaty policy (`EnableDiplomaticProposals`), the espionage mirror (`EnableEspionageMirror`), and the intel-ledger populate (`EnableIntelLedger`) — **all four static flags default `false` (`NPCDecisionProcessor.cs:42,51,62,77`), so with them off the brain DECIDES but takes no player-visible action (byte-identical).** The objective settle is always-on. Wiring/liveness note below. |
| `RelationshipState.cs` | **NEW (substrate, 2026-06-30)** The per-PAIR diplomatic record + `DiplomaticStance` enum (War/Hostile/Neutral/Friendly/Allied). A relationship is a NUMBER on a track (`RelationScore` −100..+100) that events nudge via `AdjustScore(delta)`; the headline `CurrentStance()` is DERIVED from the score by named thresholds, with `AtWar` a latched override (`DeclareWar`/`MakePeace`). Carries the treaty flags (`TradeAgreement`/`LogisticsAccess`/`MilitaryAccess`) and `LastContact`. Plain value object (not a DataBlob). Tests: `DiplomacyTests`. See `docs/society/DIPLOMACY-DESIGN.md`. |
| `DiplomacyDB.cs` | **NEW (substrate, 2026-06-30)** A faction's whole diplomatic ledger: `Dictionary<int, RelationshipState>` (other-faction Id → standing). Attached to EVERY faction by `FactionFactory` (`CreateFaction` + `CreateBasicFaction` — so all four creation paths, incl. `LoadFromJson`/`CreatePlayerFaction`). Each faction keeps its OWN view (A↔B can disagree). `GetRelationship(id)` returns existing-or-fresh-Neutral WITHOUT storing (looking ≠ a relationship); `GetOrCreateRelationship(id)` persists the row for an actual event. **First behavior wiring landed 2026-07-01:** combat `CombatEngagement.AreHostile` now reads this — a MUTUAL Friendly/Allied stance suppresses hostility (the v1 "different faction = hostile" default is otherwise preserved). Gauge `DiplomacyIffTests`. Remaining behavior slices (first-contact event, commerce/logistics gating) are `TESTING-TRACKER.md` C6. |
| `Treaties.cs` | **NEW (teeth, 2026-07-01, #33)** The treaty levers. `TreatyType` enum (NonAggression/TradeAgreement/LogisticsAccess/MilitaryAccess/DefensivePact/Peace) + `Treaties.Propose(...)` — a treaty is proposed → considered (`WouldAccept`: the target's relation score vs. a per-treaty **trust threshold** — deeper treaty needs a higher score; no ordinary treaty mid-war; Peace only ends an actual war) → on acceptance the flag is set on BOTH `RelationshipState`s + both scores warm by `SigningBonus`. Adds `NonAggressionPact`/`DefensivePact` flags to `RelationshipState`. Pure on two `DiplomacyDB`s + an entity-level overload. **v1: accept is score-gated; the design's "archetype also decides" folds in when NPC personality data exists (threshold tweak, not a rebuild). Breaking a treaty = the betrayal penalty is a later slice.** Nothing calls Propose yet → no behavior change. Tests: `DiplomacyTreatyTests`. |
| `ReactiveDiplomacy.cs` | **NEW (#35, 2026-07-01)** The reactive "Are we good?" engine — the world acts on what it OBSERVES (the developer's marquee example: a rival sees your fleet near its space → an intent probe). `ExternalStimulus` enum (FleetNearBorder / AtWarWithTheirEnemy / YouAppearWeak / CrisisOnTheirBorder / YouBrokeATreaty / …) → `Overture(obs, theirViewOfYou)` returns a `DiplomaticOverture` (AreWeGoodProbe / AllianceOffer / WarningToStop / RequestDefenseFleet / DistrustGuardRises / …), gated by their current stance; `RelationDelta(obs)` is the direct needle-nudge for the trust/distrust stimuli. Pure/stateless — the INTERNAL demand engine pointed outward (big reuse). **First live wiring landed 2026-07-02 (#35):** `NPCDecisionProcessor.RunDiplomaticDrift` now calls `RelationDelta` every monthly cycle for the stimuli computable from existing state — a **militarist neighbour** (`GovernmentDB.Militarism == High`) cools relations, a **standing treaty** warms them (kept faith) — so relationships DRIFT and cross stance thresholds on their own. Conservative: drift only, no auto-proposed treaties. **Still design/PC-gated (not invented autonomously):** `FleetNearBorder` reactions (need a territory model), NPC treaty-proposal policy, and turning an overture into a commitment. Fog cuts both ways (feed only what the observer detects). Tests: `DiplomacyReactiveTests` (table), `DiplomacyDriftTests` (the live drift). |
| `ExchangeCatalog.cs` | **NEW (#35, 2026-07-01)** The data-driven catalog of everything two factions can trade (`ExchangeDef` = key + `ExchangeCategory` [7 families] + `ExchangeKind` Instant/Standing/Event/State + `ExchangeRoute` [which existing system it drives: Ledger/Logistics/Fleets/CombatIFF/Sensors/Research/Movement/GroundCombat/People/Espionage/Galaxy/DiplomacyDB]). The catalog IS the connection map — each row is a wire into a system that already exists. Broad representative in-code set (all 7 categories); meant to GROW and can move to JSON later. `ByCategory`/`ByKey` lookups. The commitment model that EXECUTES a chosen exchange (Instant → transfer now, Standing → emit orders each cycle) is the next step. Tests: `ExchangeCatalogTests`. |
| `Diplomacy.cs` | **NEW (teeth wiring, 2026-07-01, #33)** Faction-level war ACTS: `Diplomacy.DeclareWar(aggressor, target, casusBelli, when)` / `MakePeace(a, b, when)` — latch/un-latch AtWar on BOTH ledgers (war is symmetric) + fire an `EventType.Diplomacy` event. Defensive no-ops on null/self/ledger-less. The **casus-belli → legitimacy loop is closed through here**: DeclareWar just flips the standing AtWar flag; `LegitimacyProcessor` READS it each cycle and feeds a war term gated by the government's militarism (pacifist bleeds loyalty, militarist takes pride) — so the effect sticks and compounds (derived, not a one-time nudge that gets recomputed away). Also added `DiplomacyDB.IsAtWarWithAnyone()`. Tests: `DiplomacyWarTests` + `LegitimacyProcessorTests.War_TaxesLegitimacy_ByMilitarism`. |
| `CasusBelli.cs` | **NEW (teeth, 2026-07-01, #33)** The militarism GATE on war. `CasusBelli` enum (None/BorderDispute/BrokenTreaty/AllyDefense/ConfrontRival/Retaliation) + `CasusBelliRules.WarDeclarationMoraleImpact(gov, cb)` — the one-time morale/legitimacy delta of declaring war: justified-vs-naked sets the baseline, the `GovernmentDB` **Militarism** dial swings it (High → +, Low → −). The four corners: militarist+justified = morale BONUS; pacifist+naked = regime-threatening hit. Pure/static. Wiring the delta into legitimacy/morale is #31; where a casus belli comes from (broken-treaty event, Confront-Rival demand) is the INTERNAL⟷EXTERNAL handoff. Tests: `DiplomacyCasusBelliTests`. |
| `FirstContact.cs` | **NEW (behavior, 2026-07-01)** The diplomacy "front door": `FirstContact.OnDetection(detectorFactionEntity, detectedEntity, when)` — called from `SensorScan` on a real detection. When a faction first detects an entity of another non-neutral faction, records a MUTUAL Neutral relationship row on both `DiplomacyDB`s (stamped `LastContact`) and fires a first-contact `EventType` (Neutral by default — no auto-hostility). `DiplomacyDB.HasMet` guards it to fire once per pair. Defensive/no-throw (runs in the sensor hot loop); skips neutral/own-faction targets. Liveness counter `FirstContact.ContactCount`. Tests: `DiplomacyFirstContactTests`. |
| `GovernmentDB.cs` | **NEW (substrate)** The empire-wide regime as a MODULATOR (`docs/society/GOVERNMENT-AND-POLITICS-DESIGN.md`). Four 3-notch dials (Authority/Economy/Openness/Militarism) that derive **rule overrides** (`CrewPolicy()` → the M3-2 crew rule; `Discontent()`) and **coefficient overrides** (`TaxCeiling()`/`MoraleWeight()`/`ResearchMultiplier()`/`MilitaryBuildMultiplier()`/`WarMoraleFactor()`), plus a live classifier (`Name()`/`Description()` — iconic combos table + auto-fallback). **Now ATTACHED + WIRED (2026-07-01, #30):** every faction carries a `GovernmentDB` (default all-Mid = neutral), and the coefficient dials are read by processors via `GovernmentTools` — `MoraleWeight`→migration (`PopulationProcessor`), `ResearchMultiplier`→research points (`ResearchProcessor`), `TaxCeiling`→capped billed income + morale-penalty (`ColonyEconomyProcessor`/`PopulationProcessor`). Neutral at Mid, so New Game is unchanged; the dials bite once a non-Mid regime is set (that FEEL is a PC-test). The `CrewPolicy` rule-override is now wired too (2026-07-01, #27): `ManpowerTools.ResolveBuild` reads `GovernmentDB.CrewPolicy()` at the ship-construction gate (Mid → Block, high-authority → BuildUnderstaffed/conscript). Still-deferred wire: `MilitaryBuildMultiplier` (needs a military-item classification). Tests: `GovernmentTests` (dials), `GovernmentWiringTests` (attach + lookup + tax-cap end-to-end), `ManpowerTests.ManpowerTools_ResolveBuild_HonoursHostPoolAndGovernment` (CrewPolicy end-to-end). |
| `GovernmentTools.cs` | **NEW (#30, 2026-07-01)** The single lookup the processors call to read a regime's dials. `Of(faction)` / `OwnerOf(entity)` → the entity's owning-faction `GovernmentDB`, or a shared **neutral (all-Mid) default** if none (older save / bare faction), so a read never throws or shifts the balance. This is the wiring that makes the government a real MODULATOR. |
| `MilitaryTarget.cs` | **NEW (P-3 military REACH, slice 1, 2026-07-12)** The "which enemy world do I aim at?" perception — the first of the three helpers `ConquerResolver` deferred (MilitaryTarget / MilitaryComposition / MilitaryReach). Candidates = colonies owned by a faction this one holds an `AtWar` latch toward (the latch a 3.4b coalition sets), so the chain composes: **3.4b declares the war → this scores + names the target world → the reach slice sails the fleet there**. **`BestEnemyTarget(faction)` is the developer's "best option given the circumstances" — the FIRST real UTILITY decision in the brain:** every candidate world gets a numeric SCORE (`value × reach` — population as the prize size, discounted if we'd have to jump to reach it), and the AI picks the highest, so a smaller REACHABLE world can beat a bigger DISTANT one. Named/tunable weights (`ValueWeight`/`ReachInSystem`/`ReachDistant`); the factor list is meant to GROW (weakness/garrison, true jump-route distance) — that growth is how the hardcoded menu becomes a brain "aware of its options." `NearestEnemyColonyBody(faction)` is the simple v1 candidate helper/fallback. Pure read, no warp/order surface (the move-order emission that USES the target is a separate small slice). No war / no rival colony → invalid target. Gauge `MilitaryTargetTests` (scored: a near 500k prize beats a distant 1M one). |
| `MilitaryComposition.cs` | **NEW (P-3 military REACH, slice 2, 2026-07-12)** The "do I have a strike group ready to sail?" perception — the SECOND deferred helper. `ReadyStrikeFleet(state)` returns the faction's owned fleet with the most armed hulls IF it meets `StrikeGroupMinWarships` (3, the developer's "move as a mass fleet" — don't trickle), else `InvalidEntity` (keep massing); `WarshipCount(fleet)` counts ships with a real `ShipCombatValueDB.Firepower > 0`. Pure read. **WIRED into `ConquerResolver` (the "do"):** a new Rung 1 — when `MilitaryTarget.BestEnemyTarget` names a scored enemy world AND `ReadyStrikeFleet` is massed AND not already sailing (`FleetIsMoving` guard, reads `WarpMovingDB`) AND `MilitaryReach` says the fleet can reach it (slice 3, below), it emits a **`MoveToSystemBodyOrder`** (the player's own move lever, which guards the warp landmines) to sail the fleet at that world (`Kind="StrikeFleet"`). So the reach chain is closed: 3.4b DECLARES war → MilitaryTarget SCORES the world → MilitaryComposition MASSES → the strike SAILS. Byte-identical off (runs only inside the gated `EmitOrders`) AND for any faction not at war (no target → falls to the build rung). Gauges: `MilitaryCompositionTests` (a 3-hull fleet reads ready; Conquer with a war-target + mass fleet in the fleet's OWN system decides `StrikeFleet`) + `ConquerResolverTests` (default faction, no war → still `QueueWarship`, the byte-identity tripwire). |
| `MilitaryReach.cs` | **NEW (P-3 military REACH, slice 3, 2026-07-12)** The "can my fleet actually GET THERE?" perception — the THIRD and last deferred helper, which replaces MilitaryTarget's coarse near/far PROXY with a real read off the jump-point network + the fleet's fuel/range. `Assess(fleet, targetBody)` returns a `ReachResult` = a routing `ReachTier` (**SameSystem** = a direct warp / **OneJump** = one KNOWN, discovered jump link / **Unreachable** = farther or behind an undiscovered gate) + `Hops` (0/1/-1) + `HasRange` (does a fleet ship hold enough STORED energy to spin a warp bubble — the exact `WarpMoveCommand` gate: `WarpAbilityDB.MaxSpeed>0` AND `EnergyGenAbilityDB.EnergyStored ≥ BubbleCreationCost`); derived `IsReachable` / `IsReady` (reachable + range) / `ReachFactor` (1.0 / 0.5 / 0 — the scoring discount mirroring MilitaryTarget's weights). `TierFromSystem`/`AssessRoute` expose the routing half without a fleet. **CONSUMED by `ConquerResolver` Rung 1:** the strike now sails ONLY when the target is a SameSystem warp the massed fleet has the range for — a one-jump/unreachable target or a drained fleet falls through to the build rung (keeps massing). BOUNDED v1 (labelled): a hop COUNT over the known jump graph, not a full A* multi-jump auto-router (deferred), and a warp-bubble-sufficiency check, not a full route delta-V budget; both IMPROVE on the binary in-system/distant proxy. Pure read, no warp/order surface. Gauge `MilitaryReachTests` (same-system reads reachable+cheap+ready; a drained fleet reads not-ready; a known one-jump reads reachable-but-costlier; an undiscovered gate reads unreachable). |

### The NPC brain — objective / planner / crisis / economy layer

These files make up the decision engine `NPCDecisionProcessor.Tick` drives (design refs `docs/AI-*.md`). **All are built and wired into the Tick; the order-emitting parts are gated off by default (see the processor row above). Runtime behaviour with the gates on is unverified — CI can't run the client.** Objective-settling and the read-only gauges are always-on.

| File | Purpose |

| `ThreatAssessment.cs` | **Built (F-B1b).** The fog-limited estimate of a RIVAL's military strength (the enemy-side eyes) + `GreatestThreatTo` / `IsRising` trend read (over the intel ledger). |
| `NeedsLadder.cs` | **Built (Phase 2.2).** Reads a faction's own gauges (war, morale, legitimacy, treasury, rebellion) and settles the lowest unmet `NeedTier` (Maslow rung) the brain must attend first. |
| `ObjectiveSelector.cs` | **Built (Phase 2.4a).** Given tier × `DoctrineVector` × `PersonalityDB`, names the one concrete `StrategicObjective` to pursue this cycle. **+ WAR FOOTING (2026-07-14):** `SelectWithReason(tier, doctrine, personality, atWarAndWinning, homelandInRebellion=false)` — a faction AT WAR and militarily AHEAD that is military-led/aggressive picks **Conquer** from the **Stabilize tier AND the Survive tier** (the latter only when its homeland is NOT in open rebellion). Before this, `Conquer` was reachable ONLY from `Ambition` (thriving+rich+dominant), so a **battered-but-strong belligerent never attacked** — the DevTest "phony war." **Why Survive too:** a faction on HOSTILE worlds (Mars/Venus) carries a permanent `conditions` morale penalty (`ColonyCost×12`, up to −40) that can pin it at Survive forever, so gating Conquer above Survive means it could NEVER invade (UMF). A dominant, warlike, WINNING power presses its war despite domestic strain; only a LOSING war (excluded by `atWarAndWinning`) or an actual REBELLION keeps it on defense. `WantsWar` (military-led OR aggressive) shared with the Ambition aim; `NeedsLadder.WarStanding`/`InRebellion` (now public) feed the Tick. Byte-identical for any faction not (at war and winning). Gauge: `ObjectiveSelectorTests.WarFooting_*`. |

### Espionage — the hidden-information engine (`docs/society/ESPIONAGE-AND-INTELLIGENCE-DESIGN.md`)

The spy game lives here alongside diplomacy (its outward twin). **Pure logic already written (no live consumer until E3):** `InformationLedger.cs` (the per-rival, per-facet Inferred→Confirmed→Stale intel ledger — LIVE + updated monthly as of Phase 3.1), `CovertActionCatalog.cs` (the data-driven menu of covert ops + their base detection risk), `CovertRisk.cs` (the deterministic Clean/Traced/Caught + suspicion resolver), `IntelAssessment.cs` (Confirmed intel → the TRUE number vs the fog estimate). The **cradle-to-grave BUILD** (E1→E6) turns that logic into a reachable system:

| File | Role |
|------|------|
| `IntelDirectorateDB.cs` | **NEW (Espionage E1, 2026-07-12)** The CAPACITY SEAT — the ability blob an `IntelDirectorateAtb` installs on a colony (the GeoSurvey/ResearchAcademy ability-blob pattern). Two ADDITIVE numbers: `OpCapacity` (concurrent covert ops the faction can run — the scarce budget E3's op processor spends) + `CounterIntelRating` (defensive shield the detection roll reads when the mirror runs ops against you, E5). No colony carries it until a directorate is built → additive/byte-identical. |

## FactionDataStore — the locked/unlocked split

`FactionInfoDB.Data` holds two parallel inventories:

- **`CargoGoods`** — items available to the faction right now (mineable minerals, buildable materials, unlocked component designs).
- **`LockedCargoGoods`** — items known to exist but not yet available (locked behind unresearched techs).

`Unlock(id)` moves an item from LockedCargoGoods to CargoGoods.
`IncrementTechLevel(tech)` increments the level, then calls `Unlock()` for each entry in `tech.Unlocks[tech.Level]`.

**IndustryDesigns sync gotcha:** `FactionInfoDB.IndustryDesigns` (the "things we can build" list) is populated at startup by `SetIndustryDesigns()`. When a `ProcessedMaterial` is unlocked mid-game via tech research, `Unlock()` moves it to `CargoGoods` but `IndustryDesigns` is NOT updated automatically — so colonies can't queue the material even though it's now available. **Fix in place (`ResearchProcessor.cs`, this branch):** after `AddTechPoints` causes a level-up, `DoResearch()` iterates `tech.Unlocks[tech.Level]` and syncs any materials into `IndustryDesigns`.

## NPC Doctrine

`DoctrineVector` has four float weights. They are relative, not normalized — whichever is largest drives that cycle's decision. Defaults to all zeros (no decision is made).

`NPCDecisionProcessor` registers via the standard `IHotloopProcessor` auto-discovery. It targets entities with `FactionInfoDB`. **The GlobalManager wiring gap is FIXED (keystone, 2026-06-30):** `MasterTimePulse.SimulateTimeUntil` now processes `GlobalManager.ManagerSubpulses` after the per-system block, so faction-level processors fire on their schedule (NPCDecisionProcessor: `FirstRunOffset` 5d, `RunFrequency` 30d). It still no-ops on player factions (`IsNPC == false`). Its `Tick` decision body is **built, not a TODO** (see gotcha #3): every cycle it settles a strategic objective and — behind the default-off gates — runs the order planner, treaty policy, and espionage mirror. The *loop runs* (the prerequisite every faction-level autonomous system was waiting on), and the decision layer on top of it is written and wired, just gated off by default (runtime behaviour with the gates on is unverified — CI can't run the client). Liveness gauge: `NPCDecisionProcessor.TickCount` (static; climbs each ProcessManager call) — asserted by `FactionEconomyTests.FactionLevelProcessors_FireOnceGlobalManagerIsIterated`.

**Phase 3.3 — NPC treaty policy (the Ecosystem, 2026-07-12).** `NPCDecisionProcessor.RunTreatyPolicy(faction)` is the second live behaviour the monthly `Tick` can take (after `RunDiplomaticDrift`): for the first met, not-at-war rival whose relation score clears the NonAggression bar (`RelationshipState.HostileThreshold` = −25), it calls `Treaties.Propose(faction, target, TreatyType.NonAggression, now)` — turning the built-but-uncalled proposal machinery into live diplomacy (an NPC that actively seeks détente, not just drifts). Gated by the new **`EnableDiplomaticProposals`** static (default false — a *sibling* of `EnableOrderEmission` so combat/economy-order tests and diplomacy tests can be flipped independently). Guards: skip-if-`AtWar`, skip-if-already-`NonAggressionPact` (prevents monthly re-warm churn — the #1 gotcha), skip self + `Game.NeutralFactionId`. One proposal per cycle. `Propose` signs BOTH ledgers, so the pact is two-sided. It auto-resolves through `WouldAccept` rather than surfacing an offer the player answers (a later refinement). **Four passes now (3.3 → 3.4 coalitions, 2026-07-12), each returns on first action (one treaty move per cycle):** **Pass 0 (3.4b — join a pact-ally's war):** if a faction we hold a `DefensivePact` with is `AtWar` with X and we aren't, `Diplomacy.DeclareWar` on X (casus belli `AllyDefense`) — two pacts vs a shared enemy become one COALITION whose guns actually fire (via the 3.4a combat war-override in `CombatEngagement.AreHostile`). **Pass C (3.4c — the crack):** once the shared threat is gone (`ThreatAssessment.GreatestThreatTo == -1`), a LOW-Honour faction `Diplomacy.BreakTreaty`s the pact (gated by `Treaties.WouldKeepFaith` vs `PactAbandonTemptation` 0.5 — high-Honour keeps faith); breaking re-arms the former allies. Mutually exclusive with Pass 1 (needs no threat vs needs a threat) — "alliance forms vs a riser, cracks when it dies." **Pass 1 (3.2/3.3b sharpening):** if `GreatestThreatTo` names a feared rival, propose a **DefensivePact** to a TRUSTED (Allied-75) neighbour who isn't that threat. **Pass 2 (3.3):** the plain NonAggression détente. Still byte-identical off (the whole method only runs when `EnableDiplomaticProposals` is on). Gauge: `NPCTreatyPolicyTests`.

**B6-a — the OPPORTUNISTIC declare-war trigger (2026-07-15, the developer's Q2).** `NPCDecisionProcessor.RunWarPolicy(faction)` runs right after `RunTreatyPolicy` under the SAME `EnableDiplomaticProposals` gate. Where Pass 0 above is a war of OBLIGATION (join a pact-ally's war), this is a war of CHOICE: a warlike, ambitious faction that OUT-MUSCLES a rival it already dislikes seizes the moment and declares (`CasusBelli.ConfrontRival`). Three GATES + a rank — **appetite** (`0.5·Aggression + 0.5·Ambition` from `PersonalityDB`, must exceed `WarAppetiteFloor` 0.55 so a neutral/absent personality is a no-op), **out-muscle** (`FactionRollup.MilitaryStrength(self) > MilitaryStrength(rival) × WarOutMuscleRatio` 1.25 — a clear edge, not a coin-flip), **low relation** (`rel.RelationScore ≤ HostileThreshold` −25) — and among all rivals that pass, it picks the highest opportunity SCORE (`appetite × weakness × how-hated`). One war/cycle, defensive/no-throw. **v1 uses the TRUE `MilitaryStrength` for both sides** (consistent combat-value units) gated on `HasMet`; the fog-limited `ThreatAssessment.DetectedStrengthOf` (signal-kW, a different unit) is the flagged Phase-A refinement — reconciling the two units is a Phase-A tuning job. All constants FLAGGED. Gauge: `NPCWarPolicyTests` (a warlike faction declares on a weak hated rival; a neutral personality and a warm relation each DON'T — every gate bites).

**Phase A-2a — the shared cross-domain SCORER (2026-07-15, the developer's Q1 "best not first").** `DecisionScorer.cs` (`DecisionScorer` / `PersonalityWeights` / `IScoredOption` / `DecisionFeature`) is the pure utility scorer: an option declares how much of each cross-domain `DecisionFeature` it offers (MilitarySolve / EconGain / TechGain / ExpansionGain / RiskLevel / Ruthless / Covert / Cooperative / Doctrinaire), `PersonalityWeights.Of(personality, feature)` maps each feature to a 0..1 trait blend (the ONE trait→feature map — a one-line edit per feature), and `DecisionScorer.Score = Σ feature.value × weight`; `PickBest` takes the highest. Pure/stateless, matches `docs/ai/AI-PERSONALITY-IMPLEMENTATION-SPEC.md §3. **Additive + UNWIRED (no caller) → byte-identical**. Gauge: `DecisionScorerTests` (the fingerprint worked-proof + neutral/null safety). **Phase A-2b (2026-07-15) WIRES it live:** `ConquerResolver`'s warship-mass rung (Rung 3) now SCORES the buildable warships (`PickWarship` → each wrapped as a `WarshipOption` featuring `MilitarySolve = WeaponMountCount`, a coarse firepower proxy) and masses the winner — "best not first" (the strongest hull, not whatever's first in dictionary order) instead of the old first-match. ## Gotchas

1. **`FactionDataStore` is not `FactionInfoDB`.** `FactionInfoDB.Data` is the DataStore. Code that gets the data store via `factionInfoDB.Data` and then calls `Data.CargoGoods[id]` is correct. Don't confuse the two.

2. **`IndustryDesigns` must be refreshed after any mid-game unlock.** The startup path (`SetIndustryDesigns`) and the startup-item path (`ColonyFactory.cs:43`) both do this correctly. Any new unlock call site must also update `IndustryDesigns` — either by calling the sync loop or by calling `SetIndustryDesigns()`.

> **AI ON in a real game (2026-07-15).** The four action gates (`EnableOrderEmission`/`EnableDiplomaticProposals`/`EnableEspionageMirror`/`EnableIntelLedger`) still default **`false`** so the ENGINE test suite stays byte-identical (a game built via a factory is unchanged). But every **menu-started game** now turns them ON — the DevTest button already did (`NewGameMenu.cs:925-928`), and `CreateGameCore` now does the same (so a normal New Game has living opponents). The acting path is proven in CI by **`NPCActingSensorTests`** — it opens the gates on the real DevTest sandbox, advances ~5 monthly cycles, and asserts an NPC emitted a REAL order (`StrategicObjectiveDB.LastActionKind` ≠ `""`/`"None"`). This is the gauge the whole B5→Phase-D build stacks on: "the AI acts" is now green-or-red without the SDL client.

3. **NPCDecisionProcessor FIRES (GlobalManager wired, 2026-06-30) and its Tick now SETTLES a strategic objective (Movement II Phase 2.4b, 2026-07-11) — and ACTS on it via the gated planner (2.4c+).** The loop runs; `Tick` → `UpdateStrategicObjective` reads the needs-ladder (`NeedsLadder.AssessTier`), picks an objective (`ObjectiveSelector` over doctrine + `PersonalityDB`), commits it through the hysteresis engine (`ObjectiveTransition`), and stores a `StrategicObjectiveDB` on the NPC faction; behind default-off gates (`EnableOrderEmission`/`EnableDiplomaticProposals`/`EnableEspionageMirror`) it emits real orders (the means-ends planner, treaty policy, the espionage mirror). Byte-identical with the gates off. Gauge `NPCObjectiveTickTests`. **The full AI design suite (`docs/AI-*.md`, the "quark→supercluster" design) is this processor's design reference — enter via `docs/ai/AI-BRAIN-BUILD-TRACKER.md` then `docs/ai/AI-PERSONALITY-IMPLEMENTATION-SPEC.md`; the seat-durability prerequisite it flags is already fixed (`AdminSpaceProcessor.ReconcileSeats`, 2026-07-09).**

4. **Every material a *starting* `ComponentDesign` needs must be in the colony blueprint's `StartingItems`.** `ColonyFactory.CreateFromBlueprint` unlocks `StartingItems` into `CargoGoods` first, then builds each `ComponentDesigns` entry — and `ComponentDesigner` (`Engine/Components/ComponentDesigner.cs:63`) looks up every `ResourceCost` material in the **unlocked** `CargoGoods`. A required-but-not-unlocked material crashes New Game / Quickstart in `ComponentDesigner` (`GUID object {id} not found`). This happened when `electronics`/`ree-magnetics` were added to the starting laser weapon and to the Ship Yard / Research Lab installations (commit `28967c7`) without being added to `earth.json` `StartingItems` — fixed by adding them. When you give a starting installation/weapon a new material cost, add that material to `StartingItems` too. (Related defensive change: `CargoDefinitionsLibrary.GetAny(string)` returns null instead of throwing `Sequence contains no elements`, so the failure names the missing id.) **Separate latent bug:** `TemplateFiles/ordnance.json` references `gallicite`, which is **not defined** as a mineral or material — harmless until that ordnance design is built, then it faults the same way.
