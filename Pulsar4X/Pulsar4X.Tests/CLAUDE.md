# Pulsar4X.Tests вЂ” Testing Reference

NUnit 3 test project. Runs in CI (`.github/workflows/ci.yml`) on every push/PR, and locally via `dotnet test`.

> **CI builds the engine + tests only вЂ” NOT the SDL client.** Client (UI) bugs are invisible here; they surface only in the developer's local build / the `launch.bat` console loop. See root `CLAUDE.md` в†’ "The Visibility Gate" and the CI note.

> **CI is SHARDED across 4 parallel jobs (2026-07-03; 4th 2026-07-10)** вЂ” wall-clock в‰€ the slowest single shard, not the sum (~33 min в†’ ~13 min). The suite **cannot** run in-process-parallel: several tests share static counters (`SensorScan.ScanCount`, `CombatEngagement.TickCount`, `NPCDecisionProcessor.TickCount`), so only process-level sharding is safe. Split by measured fixture time: **stations** = `StationFactoryTests` (the ~11-min bottleneck, alone), **economy** = `ScenarioHarnessTests`+`ProductionBuildTests`+`MoraleTests`, **economy-readout** = `EconomyReadoutTests` alone (a ~7.5-min game-year sim, its own shard so it can't push `rest` past stations), **rest** = the complement filter (a NEW fixture lands here by default). A red on any shard reds the run; each shard publishes its own TRX report. The root cause of the slowness is that `CreateWithColony()` re-parses ALL the mod JSON every call and the heavy fixtures call it per-test вЂ” the deferred deeper fix is to cache that parse (needs `GameFactory.CreateGame` verified read-only on the `ModDataStore`). If a new fixture becomes a heavy outlier, rebalance the shard filters in `ci.yml` (read the TRX duration column). **`EconomyReadoutTests.Economy_BaselineReadout_OverOneYear` is RE-ENABLED (2026-07-10)** вЂ” the storage-cap that quarantined it is fixed. Root cause was the `warehouse-facility` template clamping Storage Volume to 10,000 mВі (`storage.json` `MaxFormula`), so 10 warehouses held only 100,000 mВі and the 50MГ—17-goods stockpile overflowed вЂ” most minerals clamped to 0, jamming mining + refining. Fix: dropped the start stockpiles to 1M each (verified 62% of the 100k mВі cap, fits with headroom). Detail: `GameEngine/Industry/CLAUDE.md` в†’ ALPHA stockpiles.

## The Scenario Harness вЂ” `TestScenario` (build feature tests on this)

`TestScenario` is the reusable engine-level mid-game state. It stands up a REAL faction + colony the way the live game does, so feature tests (economy, combat, ground) don't re-derive a start each time. **Engine-only в†’ runs in CI.** It is the engine's *gauge board*: build a known state, advance the clock, read/assert.

### Use it
```csharp
var s = TestScenario.CreateWithColony();        // base mod, New Game wizard defaults
// handles: s.Game, s.Faction, s.Species, s.StartingSystem, s.StartingBody, s.Colony
s.QueueProductionJob("space-crete", repeat: true); // optional: give the factory/refinery work to do
s.AdvanceTime(TimeSpan.FromDays(365));           // step granularity defaults to 1 game-day
```

`QueueProductionJob(designId, count, repeat)` is the **job lever**: the factory/refinery do nothing until a job is queued. It looks the design up in the faction's unlocked `IndustryDesigns` (refined materials like `space-crete`/`stainless-steel`, and component/installation designs), builds an `IndustryJob`, and adds it to the production line that handles its industry type. A "refining" material needs its mineral inputs present (mined or in starting cargo) or the job sits at `MissingResources` вЂ” check the job Status in the industry readout.

### Feature-test pattern (read в†’ advance в†’ read в†’ assert the delta)
```csharp
var s = TestScenario.CreateWithColony();
// read STARTING gauges off s.Colony (cargo, installed components, deposits) ...
s.AdvanceTime(TimeSpan.FromDays(365));
// read ENDING gauges and assert/report the deltas (X mined, Y refined, Z built) ...
```
*(The exact cargo / mineral read API is established by the first economy readout test вЂ” link it here when it lands.)*

### How it's built (mirrors `NewGameMenu.CreateGameCore`)
load mods в†’ pick New Game defaults (first `CanStartHere` body / first species / first colony) в†’ `GameFactory.CreateGame` в†’ `StarSystemFactory.LoadFromBlueprint` per enabled system в†’ find the body by matching `GetDefaultName()` to the blueprint в†’ `FactionFactory.CreateBasicFaction` в†’ `SpeciesFactory.CreateFromBlueprint` в†’ `ColonyFactory.CreateFromBlueprint` в†’ **`startingSystem.IncrementExternalObserver(priority: true)`** (promote out of Stasis вЂ” see gotcha 6). A build failure logs the full exception (`ex.ToString()`) to the test output, then rethrows.

### Extend it
- **Expose another entity:** capture it in `CreateWithColony` and add a `public ... { get; private set; }` handle.
- **Add a precondition helper:** e.g. an `AddMine(s.Colony)` that attaches a mine component before `AdvanceTime`, so the economy actually does work to measure.
- **Use the verified namespace map below** when referencing engine types вЂ” this is where blind edits go wrong.

## Gotchas (hard-won)

1. **`CreateWithColony(base + testing mod)` throws** вЂ” the testing mod ships incomplete Armor/Theme data and NREs in colony build (it adds no species/colony). That variant is `[Ignore]`'d in `NewGameStartSmokeTests`. **Base mod alone is the working baseline.** Re-enable once the testing-mod data is fixed.
2. **`AdvanceTime` is single-threaded and sets `Ticklength`.** It sets `Game.Settings.EnforceSingleThread = true` (so a processor throw surfaces on the test thread, not the thread pool) and `Game.TimePulse.Ticklength` (default 1 day; the per-system scheduler catches up sub-daily processors within each step). Pass a smaller `step` for fine-grained needs (e.g. combat closing).
3. **Faction entities live in the `GlobalManager` вЂ” which `MasterTimePulse` NOW iterates (keystone fixed 2026-06-30).** `SimulateTimeUntil` processes `GlobalManager.ManagerSubpulses` after the per-system block, so `AdvanceTime` fires **faction-level** processors (NPCDecisionProcessor, and the future politics/diplomacy loops) as well as the colony ones. Gauge: `FactionEconomyTests.FactionLevelProcessors_FireOnceGlobalManagerIsIterated` (asserts `NPCDecisionProcessor.TickCount` climbs). *(Historical note: this used to be the "GlobalManager not iterated" trap вЂ” faction processors never fired, so things like `ColonyEconomyProcessor` were deliberately run colony-side. That workaround still works; new faction-level work can now run faction-side.)*
4. **A large chunk of the integration suite is commented out** (`/* ... */`): `SystemGenTests`, `SerializationManagerTests`, `FactoryTests`, `MiningTests`, `SavingAndLoadingTests`, parts of `PathfindingTests`. So the "all green" count is mostly unit-level. Re-enable these one at a time, CI-verified вЂ” they likely rotted (like the in-code `DefaultHumans` start).
5. **CI cannot test the client.** Engine-only. UI verification is the developer's local build.
6. **A star system defaults to `Stasis` and Stasis systems are NOT processed.** `StarSystem.ActivityState` starts at `Stasis`; `MasterTimePulse.SimulateTimeUntil` filters `Stasis` systems out of the processing loop, so a colony in a Stasis system does **zero** mining/population/industry no matter how far you advance the clock вЂ” and nothing throws, so a naive smoke test passes while measuring a frozen universe (this is exactly what hid "the mine does nothing" вЂ” the rates were correct; the system was asleep). The live game promotes a system via faction-entity detection (`Game` setup loop в†’ `UpdateActivityState` в†’ Background) and via the client observing it (в†’ Foreground). `CreateWithColony` reproduces the "player is here" case with `IncrementExternalObserver(priority: true)` в†’ Foreground (full speed; Background throttles hotloop frequency 10Г—). The economy readout prints `s.StartingSystem.ActivityState` as its first gauge for this reason. **If a feature test advances time and sees no state change, check ActivityState first.**

7. **Gauging a BUILDв†’raise/install through the industry queue is flaky in `CreateWithColony` вЂ” drive `OnConstructionComplete` directly (2026-07-06).** Two traps bit the slice-A ground-unit fielding gauge in a row: (a) zeroing a design's `ResourceCosts` to "isolate the mechanic" throws `resources can't cost 0` вЂ” use a **minimal non-zero** cost of a bulk-stocked material (`{ "stainless-steel": 1 }`) instead; (b) even then, `QueueProductionJob` + `AdvanceTime` did **not** reliably complete an `installation-construction` job in-harness (no green test drives that path to completion), so the raise never happened and `HasDataBlob<GroundForcesDB>` was False. **Fix pattern:** assert the design is a real buildable (registered on the faction, valid `IndustryTypeID`, emergent stats), then call the design's `OnConstructionComplete(colonyEntity, storage, "line", new IndustryJob(faction, id), design)` directly вЂ” the exact call `IndustryTools.cs:214` makes at completion вЂ” for a deterministic raise. The generic queue itself is covered by `ProductionBuildTests`; don't re-litigate it in every feature gauge. (See `Industry/CLAUDE.md` gotchas 6-7.)
