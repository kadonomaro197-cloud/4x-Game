# Living Galaxy — the Endogenous-Story / Event Engine (design)

**Status:** concepts + path forward (design conversation 2026-07-07). No engine code yet. A **north-star-level commitment** — it sits with `docs/NORTH-STAR-VISION.md` as a bar the whole game is measured against.

---

## The commitment, in one line

**The galaxy must be a rich, detailed, evolving story even ALONE.** If a player meets no alien faction for 250 in-game years, those 250 years must still be full of rebellions, scientific discoveries, difficult choices, lost colonies, natural disasters, and civil upheaval — a complete, complex story that **doesn't need an enemy to be interesting or fun.**

This is the internal twin of the opening question of this whole design arc ("how much does full NPC AI cost?"). The answer has two halves: **NPC factions fill the galaxy from the outside; the endogenous-story engine fills it from the inside.** Together, even in total solitude, you are always playing a game.

## Why it matters

- **It's the "every layer a complete game" bar, applied to time itself** — the downtime between big external events must be alive, not empty clock-advancing.
- **It's the payoff for all the depth we've built.** We have designed rebellions, exploration finds, disasters, demands, crises. But the current-state survey (2026-07-07) found they mostly **flip state silently** — rebellion publishes *nothing*; a survey emits a one-line log entry; a disaster applies damage and moves on. The simulation is producing drama that the player never *experiences*. This engine is what makes the depth felt.

---

## The two halves: the drama EXISTS; the surfacing does NOT

**Half 1 — the drama producers (exist or are designed):** the story is *already* being generated by systems we've mapped —
- **Rebellions / civil upheaval** ← legitimacy/rebellion + the demand engine + blocs + schism.
- **Scientific discoveries + hard choices** ← the exploration field-sites (ruins/anomalies/biospheres) + their event chains (`docs/EXPLORATION-CONTENT-DESIGN.md`).
- **Lost colonies** ← the settle-loop + starvation + rebellion + disaster.
- **Natural disasters** ← space hazards + planetary environments (solar flares, ion storms, plagues, supervolcanoes).
- **Bigger arcs** ← the crisis frontier (systemic crises: plague / AI-revolt / resource-collapse — the parked A2 mid-game pressure in `docs/DIPLOMACY-DESIGN.md`).
- **Influence / faith upheaval** ← schisms, conversions (`docs/INFLUENCE-PILLAR-DESIGN.md`).

**Half 2 — the surfacing (ABSENT).** Current state (grounded): `EventManager`/`Event` is a **fire-and-forget notification bus** whose only player surface is `GameLogWindow` (a scrolling log), plus an Aurora-style "pause the clock when this event type fires" toggle (`FactionEventLog._haltsOn`). There is **no choice/decision event, no branching consequence, no data-driven event catalog, and no paced/random event injector.** Rebellion (`LegitimacyProcessor.UpdateRebellion`) flips `IsRebelling` and opens an unused `ReactionWindowEnds` clock and publishes **nothing** — the design *anticipated* a player-response window and never built it.

**So the work is not "generate drama" — it's "surface and pace the drama the sim already makes, as choices."** This is the Visibility Gate applied to the entire simulation.

---

## What to build — the EVENT ENGINE (net-new, rides the existing bus as transport)

Three capabilities, none of which exist today:

1. **Choice events.** An event that can PAUSE and present the player **options**, each with **consequences routed back into the sim.** (Today an `Event` carries only a pre-formatted message; the `_haltsOn` toggle can pause the clock but shows no options.) A choice event = trigger + description + options[] + per-option consequences.
2. **A data-driven event catalog.** Events defined in JSON (like the exchange / covert-action / field-site catalogs), so new events are content, not code. (Today every event is a hardcoded `Publish(...)` call site; no event JSON exists.)
3. **Triggers + a pacer.** A layer that (a) **watches sim state** and fires the matching event (legitimacy entering the collapse band → the rebellion event; a survey completing → the discovery event; a hazard rolled → the disaster event), and (b) **injects paced endogenous events** on a cadence so there is always a live thread even when the sim is quiet. (Today nothing generates events spontaneously.)

The existing `EventManager` bus and the `_haltsOn` pause are the **transport** — reuse them; the choice/consequence/catalog/pacer machinery is the build.

## The shape of an event

> **trigger** (a sim-state watcher OR the pacer) → **event** (description, from the catalog) → the player is **presented choices** (the clock pauses, reusing `_haltsOn`) → the chosen option's **consequences route back into the systems** (adjust legitimacy, spawn a rebellion/fleet, grant tech, shift a relation, damage a colony) → the world moves on, changed.

Data-driven and broad, exactly like the other catalogs. A delegate can auto-resolve low-stakes events to a policy (the same "play at your own altitude" valve — you don't get a popup for every minor incident unless you want to).

## The firewall (so it's "something to do IN DETAIL," not popup spam)

- **Every event is a CHOICE THAT MATTERS** — consequences that stack into the systems, never a flavor popup with one "OK" button.
- **Paced by significance** — a cadence + a significance threshold so the player gets *meaningful moments*, not noise. Minor stuff goes to the log (the existing bus); only real decisions interrupt.
- **Delegation valve** — auto-resolve trivial events to a stance; surface only what needs the player (anti-"feels like a job", consistent with the whole delegate model).

## The first slice (cheapest proof — a hook already exists)

**Wire the unused rebellion reaction-window into a choice event.** `RebellionDB` already has `ReactionWindowEnds` / `WindowExpired(now)` — the design already anticipated "a colony is rebelling; here is your window to respond." The first event: *"[Colony] is in open revolt. [reaction window]"* → choices: **pour in legitimacy** (enact demands / ship aid), **send troops** (ground suppression), **negotiate** (concede something), **let it go** (it secedes — feeds the influence/secession model). That single event proves the whole engine (trigger from sim state → pause → choices → consequences back into legitimacy/rebellion/ground/influence) on a drama-producer that already has the hook and currently does nothing visible.

## Connections (Prime Directive)

- **The notification bus** (`EventManager`/`Event`/`GameLogWindow`, `FactionEventLog._haltsOn`) — the transport to reuse; extend `Event` with options/consequences.
- **Legitimacy / rebellion** — the first and richest producer (and the unused reaction-window hook).
- **Exploration field-sites** — discovery events + anomaly event-chains are catalog entries.
- **Hazards / environments** — disasters become events, not silent damage.
- **Internal politics (demands) + Influence (schism/conversion)** — upheaval events; the reactive-demand engine is a sibling generator (both turn sim state into a player-facing ask).
- **The crisis frontier** (`docs/DIPLOMACY-DESIGN.md` A1/A2) — multi-stage crisis arcs are the large-scale event chains.
- **Delegation** (`docs/AI-SELF-PLAY-DESIGN.md`) — auto-resolve valve for low-stakes events.
- **The GlobalManager iteration** (done) — the pacer/watchers run as processors that must actually fire.

## Locked vs. open

**Locked (2026-07-07):**
- **The commitment:** the game is a rich, detailed, evolving story SOLO — no enemy required. A north-star bar.
- **The drama already exists in the systems; the missing piece is the SURFACING** — an event engine, not more drama generators.
- **Build a narrative-event engine** = choice events + a data-driven catalog + triggers/pacer, riding the existing notification bus as transport.
- **Events are choices that matter, paced by significance, with a delegation auto-resolve valve** (not popup spam).
- **First slice = the rebellion reaction-window choice event** (the hook already exists, currently silent).

**Open (decide when we build):**
- The pacing model (cadence + significance threshold; how often is "always something" without being noisy).
- How deep event *chains* go (single choice vs multi-stage arcs) for v1.
- The consequence vocabulary (the standard set of sim-effects an option can emit — the catalog's building blocks).
- How auto-resolve decides (a per-category stance, like the delegate stances).
- Crisis-arc authoring (scripted vs emergent) — likely its own later pass.

**Build order:**
1. **Extend the event model + the choice UI** (options + consequences; reuse `_haltsOn` to pause-and-present) — the engine spine.
2. **The rebellion reaction-window event** (the first slice; proves trigger→choice→consequence end-to-end).
3. **The data-driven catalog + a trigger table** (turn hardcoded `Publish` sites into catalog entries where they deserve a choice).
4. **The pacer** (endogenous cadence so quiet stretches still breathe).
5. **Crisis arcs** (multi-stage chains; ties to the diplomacy-doc crisis frontier).
