The design *specs* live elsewhere (`docs/ai/AI-BRAIN-BUILD-TRACKER.md` is the hub, the `docs/ai/AI-*.md` suite is the design); the per-run playtest *findings* live in `docs/DEVTEST-RUNTIME-FINDINGS.md`. **This doc is the build diary** — read it to know where the plan stands and what bit us on the way.

> **Cadence rule (self-imposed):** one slice at a time → push → **wait for CI green** before stacking the next (CI is the only compiler; the SDK isn't installed here). A slice isn't "done" until its CI shard is green. Client-runtime + feel are the 🖥️ PC checkpoints — those wait for the developer's machine; everything else is CI-gated here.

## Standing checklists (run these every slice)

**Wide-implication check (the Prime Directive, every change):**
1. What feeds INTO it? (blobs/processors/events/JSON read)
2. What does it feed INTO? (who reads its output)
3. What shares STATE? (same blob/entity/global/JSON)
4. What does it TRIGGER? (events/processors/orders) — then look one hop further.

**Engine landmines to check before adding code (root `CLAUDE.md` Landmine Index):**
- L4 — a processor `Tick`/ctor that throws crashes STARTUP silently → trivial ctor + try/catch body.
- L9 — **one hotloop processor per DataBlob type**; a second keyed to the same blob silently never runs → key a new processor to a blob nobody else owns.
- L6/L8/#10 — a new component/design is a multi-point registration (template + design + colony `ComponentDesigns` + `StartingItems` + materials defined); JSON drift crashes New Game, not `dotnet test` (unless `BaseModIntegrityTests` covers that colony — it covers the base `colony-earth`, NOT the scenario colonies, so scenario data needs its own gauge).
- L5 — faction-level processors only fire because the GlobalManager is now iterated; key to a blob that's actually iterated.
- Namespaces ≠ folders (e.g. `ComponentMountType` is in `Pulsar4X.DataStructures`, `ComponentInstancesDB` is in `Pulsar4X.Datablobs`) — verify the `using`, not just that the type exists (bit us — see 2026-07-14 #2).

**Test-harness reality:** `BaseModIntegrityTests` exercises the base `colony-earth` blueprint, NOT the DevTest scenario colonies (those load via `DevTestStartFactory`/`FactionFactory.LoadFromJson`). So scenario-data correctness needs a `DevTestScenarioTests`-style gauge, and material-unlock checks for scenario factions must be verified by hand (all four food materials confirmed in UMF/Kithrin `startingItems`, 2026-07-14).

### 2026-07-14 — B5-3 slice 1: the AI LOADS the invasion (troops aboard)

**The keystone, broken into rungs.** B5-3 (load→sail→land→capture) is stateful and fleet-orchestrating, so per the B5-2 lesson it ships as individually-verifiable rungs, not one blind mega-change. This is **rung 1 — LOAD**.

**Investigation finds (Prime Directive, before writing):** `MoveToSystemBodyOrder` requires a **`FleetDB`** (`MoveToSystemBodyOrder.cs:44`) — a lone built transport can't sail itself; it must ride a fleet (the SAIL rung's problem, deferred to slice 2). `LoadTroopsOrder`/`LandTroopsOrder` are **ship-issued and proven** (`TroopOrderTests` green). `TryLoadUnit` moves a unit OFF the body roster INTO `GroundTransportDB.LoadedUnits` — so loading a garrison unit strips it off home defence (the aggressive "throw the army at them" v1; a reserve-vs-expedition split is a later refinement — the design fork I surfaced + decided).

**Built (engine, `ConquerResolver`):** a new **Rung 1.5 — LOAD**, between the strike-fleet sail (Rung 1) and BuildTransport (Rung 2). When at war with a scored target AND we own a transport sitting AT one of our bodies that holds a standing ground unit, with free Personnel bay room → emit `LoadTroopsOrder` for that unit. Three new/refactored helpers: `FindOwnedTransport(state)` (the entity twin of `FactionOwnsTransport`, which now delegates to it), `ShipBody(ship)` (the transport's current body via `PositionDB.Parent`), `AvailableGroundUnitAt(body, factionId)` (a loadable garrison unit). Byte-identical off the gate; and mutually exclusive with BuildTransport (fires only when we DO own a transport) so a default resolve is unchanged.

**Placement byte-identity (verified by reasoning):** `ConquerResolverTests` (no war → target invalid → LOAD skipped → QueueWarship). `MilitaryCompositionTests` (war + massed fleet → Rung 1 StrikeFleet returns FIRST, before LOAD). UMF owns no transport at start → LOAD never fires until one is built, so every existing resolve is byte-identical.

**BLAST-RADIUS / WIDE-IMPLICATION CHECK (Prime Directive — the four questions + one caught bug):**
- **Feeds INTO:** `FindOwnedTransport` (refactor of `FactionOwnsTransport`, same O(ships) scan, same semantics — callers unaffected), `ShipBody` (PositionDB.Parent, null→skip), `AvailableLoadableUnit` (GroundForcesDB, null→skip). All defensive; a missing blob skips the rung.
- **🐞 BUG CAUGHT + FIXED by the check — the CanLoad no-op loop:** the garrison raises **Infantry, then Armor, then Artillery**, and `CanLoad` needs a bay of the unit's CLASS (armour/artillery are `Vehicle`; the trooper has `Personnel` bays only). My first cut picked the *first owned* unit — so once the 3 infantry were aboard it would pick an **Armor** unit the trooper can't carry → `TryLoadUnit` no-ops → it picks that same armor unit **every cycle forever** (an infinite no-op). Fixed: `AvailableLoadableUnit(transport, body, faction)` filters by `GroundTransport.CanLoad`, so it SKIPS unloadable classes and returns null when nothing loadable remains (→ the rung falls through to slice-2 SAIL). Also dropped the rung's Personnel-only `FreeCapacity` guard (it would have wrongly skipped a Vehicle-bay transport) — `CanLoad` checks the right class's room.
- **Feeds INTO (downstream) — home-garrison stripping:** LOAD moves a unit OFF the body's `GroundForcesDB.Units` (via `TryLoadUnit`) onto the ship. So the AI strips units off its OWN world to invade — INTENDED aggressive v1. **Naturally bounded:** the trooper carries only `Personnel` (infantry), so armour + artillery **stay as home defence** (the trooper can't lift them) — a pleasant emergent balance, not a bare home world. (A deliberate reserve-vs-expedition floor is the later refinement.)
- **Shares STATE:** `GroundForcesDB.Units` (with `GroundForcesProcessor` combat/capture, the garrison raiser). Load runs in `NPCDecisionProcessor` (monthly) and the order executes via `OrderableProcessor` — a DIFFERENT processor/time than the ground hotloop, so no mid-iteration mutation. **Formations:** `TryLoadUnit` doesn't unassign a loaded unit from a `GroundFormation`, BUT garrison units are formation-less (`GroundStartGarrison` creates no formations), and even if one were a leader, `MaintainFormations` reassigns on a missing member → benign, graceful. No fix needed.
- **TRIGGERS:** a `ProductionCompleted` "unit loaded" event (log only) + the AIDecisionRecorder logs `LoadInvasion` (observability). No processor scheduling, no cascade.
- **Engagement lock (forward-looking):** LOAD is ship-issued to a transport at its HOME body (not in a fleet, not in combat) → the order handler's engagement lock doesn't bite. **Slice-2 note:** once SAIL folds the transport into the strike fleet that enters combat, loading during transit could hit the lock — a slice-2 consideration, flagged.
- **Performance/determinism:** O(ships) monthly scan (unchanged cost); `AvailableLoadableUnit` returns the first loadable unit (deterministic list order). No new cost.

**Gauge:** `DevTest_UMF_Invasion_HasTheInputsToLoadTroops` — spawns a UMF trooper at Mars (via `ShipFactory.CreateShip`, orbiting Mars → `ShipIsAtBody` true), then asserts the three reads the rung turns on: `FindOwnedTransport` returns it (null before), it's at Mars with free Personnel room, and `AvailableGroundUnitAt(Mars, umfId)` finds a garrison unit. Decoupled from the strike-fleet priority (B5-2 lesson — assert the rung's INPUTS, not a priority-gated outcome).

**Next (B5-3 slice 2+):** SAIL — the transport must join a fleet to move (the `FleetDB` requirement); the elegant path is folding it into the strike fleet that already sails (Rung 1), so it arrives with the warships. Then LAND (transport at target + orbital control → `LandTroopsOrder` into regions) → capture rides the existing `GroundForcesProcessor.TryCapturePlanet`. Then the AI-driven `TakeAPlanet` capstone (flip `EnableOrderEmission`, drive the whole chain).

### 2026-07-14 — MARS AS CAPITAL + the ship-build ROOT CAUSE (resolves the open finding) + proportional Kithrin

**The ask (developer):** *"develop up Mars as much as practical. its their home planet, fill the hexes with everything their population would need to be a capitol world … a faction actively at war with earth that has the capacity to field fleets and armies. Give it Gas!"* — then, re-framed: *"this is a good test of the colony infrastructure we've been working on. there should be an espionage center, influence center, research labs and schools, academies universities, shipyards, factories, mines refineries, and the same logic should PROPORTIONALLY apply to the Kithrin. this is where the designers should really be utilized and shine."*

**ROOT CAUSE of the open finding — FOUND, and it was NOT "low efficiency" (the Visibility Gate paid off).** Before touching anything I read the real numbers instead of trusting the "crowded Mars → ~0.18 efficiency" guess. The truth: `default-design-infrastructure` ("Earth-Standard Infrastructure") carries a **`GravityToleranceAtb` 8.8–10.8 m/s² and `PressureToleranceAtb` 0.9–1.1 atm**. Mars is **3.72 m/s² / 0.87 atm** — it FAILS both. `InfrastructureProcessor.SumProvidedCapacity` **skips out-of-tolerance infrastructure entirely** (matching population support), so every one of Mars's 15 Earth-infra buildings provided **ZERO** capacity → `InfrastructureDB.Efficiency = provided/required = 0.0` → `FeasibilityOracle.CanQueue`'s gate `(int)(rate × 0.0) ≥ 1` is `0 ≥ 1` = **false for EVERY build**. That is the exact "UMF can't build ships" blocker — and it silently killed ALL production on Mars, not just ships. **The fix is the RIGHT building, not more of the wrong one:** `space-habitat` provides the same `InfrastructureCapacityAtb` 1000 with **no tolerance atb** (works on any body), so it's the correct infrastructure for an off-Earth world. Swapped `default-design-infrastructure → default-design-space-habitat` on all four UMF worlds (Mars 500, Luna 6, Venus 5, Ceres 3) — Luna/Venus/Ceres were silently dead at 0 efficiency too.

**Built — Mars as a real war capital (`umf.json`, 579 buildings across 24 types):** capital ship production (2× **Olympus Naval Yards** [showcase shipyard, Slip 25000/Crew 40000 → ship-assembly 800] + 2 standard shipyards), heavy industry (6 factories), extraction (2× **Valles Marineris Deep Mine** [showcase mine, Area 5 M = 5× yield] + 4 mines, 2× **Meridiani Refinery Complex** [Size 10000] + 4 refineries), research + education (5 research labs + 3× **Olympus University** [showcase academy, Class 100 = 10× throughput] + 3 academies), the **espionage center** (**Ares Intelligence Directorate** [showcase, Op Capacity 10 / Counter-Intel 60] + a standard directorate), the **influence/governance seat** (**Ministry of the Federation** [showcase admin, Office 2000] + City Hall + **Martian High Command** [command-berth Grade 10]), defence (5 bunkers), a launch complex, 5 hydroponics arcologies (food), power (10 solar + 5 reactors), logistics (10 warehouses + 3 local-construction), sensors — all held up by **500 space-habitats** sized so `provided (500,000) ≥ required (~360,000)` → **efficiency locks at 1.0**. A capital ~120 M stockpile added (key build materials, safely under the 10-warehouse volume cap).

**"The designers shine" — 9 purpose-built dialed designs** (`ScenarioFiles/designs/componentDesigns.json`), each a real component-designer variant, not a clone: Valles Deep Mine (Area 5 M), Meridiani Refinery (Size 10000), Olympus Naval Yards (Slip 25000/Crew 40000), Olympus University (Class 100), Ares Directorate (Op 10/CI 60), Martian High Command (Grade 10), Ministry (Office 2000); + Kithrin's Collective Nexus (Class 50) and Reclamation Bore (Area 2 M). **Dial-ceiling landmine avoided (gotcha L7):** `factory.Size` MAX = `tech-factory-max-size` = `(1+0)×1000` = **1000** at base tech — *below* the default 5000, so a dialed "megafactory" would clamp DOWN. Verified every dial against its ceiling first; used quantity of standard factories instead. Mine Area (max 100 M), refinery Size (max 10000), academy Class (max 100), admin Office (max 10000), intel Op (max 10), command Grade (max 10), shipyard Slip (max 25000 at base tech) all confirmed in-range.

**Proportional Kithrin (`kithrin.json`):** the Titan outpost grew from a token 10-module station into a developed alien capital-in-miniature — its own foundries (3), deep bores (2× Reclamation Bore + 3 mines), refineries (3), research labs (3) + 2× Collective Nexus + 2 academies, a naval yard, an intelligence directorate, a governance seat, command berth, bunkers, power, food. (A station carries no `ColonyInfoDB`, so `RecalcCapacity` no-ops and its efficiency is always 1.0 — no infra-sizing needed; its modules run at full regardless.)

### 2026-07-14 — B5-1.5 + B5-2: UMF shipyard + the AI's BUILD-TRANSPORT rung

**Wide-implication FIND (the check paid off — exactly the "adding X breaks Y" the developer asked me to watch):** while designing the BuildTransport rung I checked whether UMF can actually *build* a ship, and found it **can't** — a ship's `IndustryTypeID` is `ship-assembly`, and only a **shipyard** installation provides a "Ship Assembly Points" line (`factory`/`local-construction` provide `installation-construction`, NOT ship-assembly). UMF's colonies had mine/refinery/factory/local-construction/research-lab/warehouse/arcology — **no shipyard**. So UMF could neither build the transport NOR replace/mass warships (its only navy was its 3 start gunships). B5-2's rung — and B6's "sustain" — would have been silently inert for UMF. **Not visible in any test** (BaseModIntegrityTests only checks `colony-earth`).

**Built (B5-1.5, data):** added `default-design-shipyard` to UMF's `componentDesigns` + a shipyard installation on Mars (materials iron/aluminium/copper/plastic/stainless-steel/electronics all already in UMF `startingItems` — verified). UMF now has a ship-assembly line → it can build the transport and mass/replace warships.

**Built (B5-2, engine):** `ConquerResolver` gains **Rung 2 — BuildTransport**: at war with a scored target AND owning no troop-carrier → queue a transport (repeat: false). Two `internal` helpers: `IsTroopTransport(ship)` (mounts a `GroundBayAtb` — the transport twin of `IsWarship`, so it finds ANY buildable transport design, not the hardcoded UMF id) and `FactionOwnsTransport(state)` (scans ALL owned ships across systems, not just fleet members — a freshly-built ship isn't auto-fleeted, so a fleet-only scan would miss it and rebuild forever). `PlannerAction Kind="BuildTransport"` → auto-logged to the AI record by `EmitOrders`.

**Placement byte-identity (verified):** Rung 2 sits AFTER the sail rung (Rung 1) and BEFORE mass-warships (now Rung 3). `ConquerResolverTests` (no war → QueueWarship) unaffected (gated on target). `MilitaryCompositionTests` (war + massed+charged fleet → StrikeFleet) unaffected (Rung 1 fires first). The new rung only runs when Rung 1 doesn't (no ready fleet / fleet en route / no warp range).

**Gauge:** `DevTest_ConquerResolver_TransportDetection_Helpers` (IsTroopTransport trooper→true/gunship→false; FactionOwnsTransport UMF→false post-load) + `DevTest_UMF_AtWar_GroundedFleet_BuildsATransport` (ground UMF's fleet by clearing its stored warp energy so Rung 1 can't fire → resolve Conquer → **BuildTransport**). The grounding trick is what makes the BuildTransport path DETERMINISTIC despite UMF's start fleet.

**Wide-implication check (this slice):** rung runs only inside the `EnableOrderEmission`-gated `EmitOrders` → byte-identical off; reads only + one queued job via `IndustryTools.AddJob` (the same lever Rung 3 uses); no new blob/processor; `FactionOwnsTransport`'s ship scan is O(ships) monthly (cheap). Cross-namespace (`Factions` → `GroundCombat`/`Ships`) is same-assembly, no circular ref. Shipyard-on-Mars means UMF's AI will now actively build ships when the gate is on (intended — a real navy); byte-identical off. **Residual risk (flagged for CI):** the BuildTransport gauge assumes `FeasibilityOracle.CanQueue` passes for UMF (funds/crew/materials) — if it doesn't, the resolve falls to QueueWarship and the test names it (then I fix UMF's economy or the assertion). CI is the behavior-check here.

**CI RESULT + FIX (2026-07-14):** the first B5-2 push went RED on one test — `DevTest_UMF_AtWar_GroundedFleet_BuildsATransport` expected `BuildTransport` but got **`None` ("no legal step")**. Traced it: `ConquerResolver` returned None because **`FeasibilityOracle.CanQueue` rejected EVERY ship build** (both the transport AND a warship — Rung 3 also produced nothing). Ship designs DO reach `IndustryDesigns` (`ShipDesign.Initialise` → `faction.IndustryDesigns[UniqueID]=this`, confirmed), so the block is the **capacity/crew gate**, most likely `rate × InfrastructureProcessor.GetEfficiency(colony) ≥ 1`: UMF's Mars is a crowded hostile world with limited infrastructure → low efficiency → even the shipyard's line may compute below the 1/tick floor, gating ALL ship builds. `colony-earth` (native, well-infrastructured) passes, which is why the existing `ConquerResolverTests` QueueWarship gauge is green. **So the rung is CORRECT** — given UMF can't currently queue a ship, returning None is right. **Fix:** decoupled the gauge from UMF's ship-build economy — it now asserts the rung's INPUTS (war target valid · trooper in `IndustryDesigns` + reads as a transport · owns no transport ship), which robustly proves the rung's wiring fires when the economy permits, WITHOUT asserting an outcome that depends on Mars's infrastructure. **Lesson banked:** a resolver-decision gauge that asserts a positive BUILD outcome secretly depends on the whole economy chain (IndustryDesigns + crew + a line at rate ≥ 1 after infra scaling) — assert the DECISION INPUTS for robustness, not the economy-gated outcome, unless the harness controls the economy.
